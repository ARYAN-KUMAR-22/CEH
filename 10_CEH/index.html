FTP Security Principles & Port Hardening

Protecting File Transfer Services

Default FTP Configuration Risks

  • Standard Port: FTP uses port 21 (TCP control) and port 20 (TCP data)
  • Visibility: Network scanners quickly identify FTP on default port
  • Vulnerability: Once identified, standard FTP exploits can be applied
  • Clear Text: Credentials transmitted without encryption by default

Primary FTP Security Strategy: Port Obfuscation

Organizations avoid using default port 21 for FTP. Instead, they:

  • Configure Non-Standard Ports: Use ports like 2121, 8121, or random high-numbered ports
  • Firewall Protection: Implement firewall rules to filter traffic to custom FTP ports
  • Obscurity Benefit: Attackers conducting network scans won't see port 21 open
  • Attack Difficulty: Without knowing the FTP port, exploiting FTP becomes significantly harder

FTP Port Configuration Example

Standard (Vulnerable):
    nmap 192.168.1.50
    Starting Nmap...
    21/tcp open ftp  ← FTP IMMEDIATELY IDENTIFIED!
    
Hardened (With Port Change):
    nmap 192.168.1.50
    Starting Nmap...
    Output shows many ports open but NOT 21/tcp
    
    Actual FTP on custom port:
    nmap -p 2121 192.168.1.50
    2121/tcp open ftp  ← Only visible if specifically checking this port

FTP Port Security Layers

Layer 1: Network Scanning (Attacker Perspective)
▼
$ nmap 192.168.1.50 Starting Nmap... PORT STATE SERVICE 22/tcp open ssh 80/tcp open http 443/tcp open https 3306/tcp open mysql 5432/tcp open postgresql # Port 21 NOT VISIBLE # FTP is hidden on alternate port

Attacker cannot identify FTP service through standard port scanning

Layer 2: Firewall Filtering
▼
# Firewall Rule Example Allow Port 2121 from: 203.0.113.0/24 ONLY (Authorized subnet) Block Port 2121 from: 0.0.0.0/0 (Everyone else) # Attempt from unauthorized IP: $ ftp 192.168.1.50 2121 Connection refused (Firewall blocks) # Attempt from authorized IP (203.0.113.5): $ ftp 192.168.1.50 2121 Connected! (Firewall allows)
Layer 3: Public vs Private Network Access
▼
Server Configuration: - Public IP: 203.0.113.50 (Internet-facing) - Private IP: 10.0.1.50 (Internal network) Firewall Policy: - Bind FTP to Private IP: 10.0.1.50:2121 - FTP NOT accessible on public IP - Only internal users can access FTP - External attacks cannot reach FTP service Result: $ ftp 203.0.113.50 2121 (Public attempt) Connection refused $ ftp 10.0.1.50 2121 (Internal attempt) Connected!

✓ Multi-Layer FTP Security

  • Non-standard ports hide service from automated scanning
  • Firewall rules restrict geographic/network access
  • Private IP binding prevents public internet access
  • Combination approach makes FTP significantly more secure

Anonymous FTP Server Configuration

Setting Up Open File Share Services

Anonymous FTP Use Cases

Anonymous FTP allows unrestricted access without authentication. Common legitimate uses:

  • Public software/document distribution (like Linux ISO downloads)
  • Open-source project file hosting
  • Publicly shared resources that require no authentication

⚠️ Anonymous FTP Security Risks

  • No Authentication: Anyone can access without credentials
  • Data Leakage: Sensitive files may be accidentally exposed
  • Malware Distribution: Attackers can inject malicious files
  • Service Abuse: Large storage consumption from malicious use

vsftpd Installation & Configuration (Ubuntu)

$ apt update
$ apt install vsftpd

# Create FTP share directory
$ mkdir -p /home/ftp/public
$ mkdir -p /home/ftp/public/documents
$ touch /home/ftp/public/documents/readme.txt

# Set ownership for anonymous access
$ chown -R nobody:nogroup /home/ftp

# Edit configuration file
$ nano /etc/vsftpd.conf

vsftpd Configuration (Anonymous Mode)

# /etc/vsftpd.conf - Anonymous FTP Configuration

# Enable anonymous FTP
anonymous_enable=YES              # Change from NO to YES

# Disable local user login
local_enable=NO                   # Prevent system users from accessing

# Define anonymous root directory
anon_root=/home/ftp               # Directory where anonymous users land

# Allow write operations
write_enable=YES                  # Permit file uploads
anon_upload_enable=YES            # Anonymous users can upload
anon_mkdir_write_enable=YES       # Anonymous users can create directories
anon_other_write_enable=YES       # Allow delete/rename

# Security settings
anon_umask=0022                   # File permission: 644 (rw-r--r--)
no_anon_password=YES              # Don't prompt for password

# SSL/TLS (optional)
ssl_enable=NO                     # Unencrypted (basic)

# Logging
log_ftp_protocol=YES              # Log all FTP commands

Local User FTP Access Configuration

User-Based Authentication and Access Control

Local User FTP vs Anonymous

  • Anonymous: No authentication required, public access
  • Local User: System user credentials required for access
  • Security: Local user FTP provides accountability and controlled access
  • Use Case: Authorized staff, partner organizations, internal teams

Configure Local User FTP (vsftpd.conf)

# /etc/vsftpd.conf - Local User Configuration

# Disable anonymous FTP
anonymous_enable=NO               # No anonymous access

# Enable local users
local_enable=YES                  # System users can login
write_enable=YES                  # Allow write operations

# Restrict users to home directory (chroot jail)
chroot_local_user=YES             # Confine users to their $HOME
allow_writeable_chroot=YES        # Allow writes in home directory

# User list management
userlist_enable=YES               # Use user whitelist/blacklist
userlist_file=/etc/vsftpd.userlist
userlist_deny=NO                  # If NO: list = allowed users
                                  # If YES: list = denied users

# Permissions
local_umask=0022                  # Files: 644, Dirs: 755

# Logging
log_ftp_protocol=YES

Create FTP Users on System

# Create system users for FTP access
$ useradd -m -s /usr/sbin/nologin alice
$ useradd -m -s /usr/sbin/nologin bob

# Set passwords
$ passwd alice
New password: password123
$ passwd bob
New password: bobsecure456

# Create whitelist (users allowed to use FTP)
$ echo "alice" > /etc/vsftpd.userlist
$ echo "bob" >> /etc/vsftpd.userlist

# Change ownership of home directories
$ chown alice:alice /home/alice
$ chown bob:bob /home/bob

# Restart FTP service
$ systemctl restart vsftpd

Web Server Architecture & Deployment

Understanding Server-Side Web Application Hosting

Web Server Purpose

A web server is a software process that listens on port 80 (HTTP) or 443 (HTTPS) and responds to client requests by serving web content. It forms the core infrastructure for internet-accessible applications.

Web Server Lifecycle: Request → Response

  1. Client Request: Browser sends HTTP request to server IP:port
  2. Server Reception: Web server receives request and parses it
  3. Resource Lookup: Server locates requested file (HTML, PHP, image)
  4. Processing: Execute dynamic code (PHP, Python, Node.js)
  5. Database Query (optional): Fetch dynamic data from backend
  6. Response Generation: Build HTTP response with headers + body
  7. Transmission: Send response back to client
  8. Rendering: Browser renders HTML/CSS/JavaScript locally

Popular Web Server Options

Server OS Platform Market Share Architecture
Apache HTTP Server Linux, Windows, Unix ~40-45% Process-based (one process per connection)
Microsoft IIS Windows Server only ~20-25% Thread-based (integrated with Windows)
NGINX Linux, Unix, BSD ~15-20% Event-driven (asynchronous)
LiteSpeed Linux ~2-3% High-performance event-based

⚠️ Default Web Server Risks

  • Open to Internet: By default, web server listens on 0.0.0.0:80 (all IPs)
  • Information Disclosure: Default pages reveal server version and software info
  • Unpatched Vulnerabilities: Default installations may not include security patches
  • No Authentication: Web server serves files without access control by default
  • CGI/Script Execution: Poorly configured script execution can allow RCE

SSH Server Configuration & Installation

Secure Remote Terminal Access

SSH vs Telnet: Evolution of Remote Access

  • Telnet (Obsolete): Clear-text remote shell - credentials visible to attackers
  • SSH (Modern Standard): Encrypted remote shell - credentials and data protected
  • Difference: SSH adds encryption layer around all communication

SSH Components

  • SSH Client (ssh command): Tool for users to connect to remote servers
  • SSH Server (sshd daemon): Service running on server listening for incoming connections
  • Configuration Files:
    • ~/.ssh/config (Client-side user preferences)
    • /etc/ssh/sshd_config (Server-side settings)

SSH Authentication: Passwords vs Public Keys

Securing Remote Access with Multiple Authentication Mechanisms

Authentication Methods Comparison

  • Password Authentication: User provides password each connection
  • Public Key Authentication: Asymmetric cryptography credentials
  • Multi-Factor: Combination of password + key (most secure)

Enable Root User SSH Access

# By default, root cannot SSH in
$ ssh root@203.0.113.50
Permission denied (publickey,password).

# To allow root SSH (not recommended):
$ sudo nano /etc/ssh/sshd_config

# Find this line:
PermitRootLogin prohibit-password

# Change to:
PermitRootLogin yes

# Restart SSH to apply changes
$ sudo systemctl restart ssh

# Now root can SSH:
$ ssh root@203.0.113.50
root@203.0.113.50's password: *******
root@server:~#

Microsoft IIS Web Server Installation

Deploying Web Server on Windows Server Operating System

IIS vs Apache Architecture

  • Apache: Open-source, cross-platform, process-based architecture
  • IIS: Microsoft proprietary, Windows-only, integrated with .NET/Active Directory
  • Target Apps: Apache for PHP/Python, IIS for ASP.NET/C#

Windows Server Manager

IIS is installed through Windows Server Manager - a centralized server administration console for managing roles and features.

IIS Installation Steps (GUI)

1. Open Server Manager
   → Dashboard shows server status and managed services

2. Click "Manage" → "Add Roles and Features"
   → Role-Based or Feature-Based Installation (select Role-Based)
   → Select server from server pool
   → Check "Web Server (IIS)" from roles list
   → Click "Add Features" when prompted

3. Review IIS role description
   → Click "Next" through installation wizard
   → Verify all required role services are selected

4. Installation & Configuration
   → IIS automatically selected role services
   → Server automatically creates IIS directories and configuration

5. Complete Installation
   → Restart may be required
   → IIS service starts automatically

Result: IIS is now installed and listening on port 80

Remote Desktop Protocol (RDP)

Graphical Remote Access to Windows Servers

RDP vs SSH: Different Use Cases

  • SSH: Text-based terminal access, works on all platforms
  • RDP: Graphical GUI access, primarily Windows systems
  • RDP Use: Visual server administration, GUI-based application management

Remote Desktop Protocol Characteristics

  • Port: TCP port 3389 (standard RDP port)
  • Encryption: TLS/SSL encryption for session security
  • Compression: Screen data compressed to reduce bandwidth
  • Experience: Full GUI desktop environment over network
  • Client Availability: Windows, macOS, Linux clients available

RDP Connection from Windows Client

2. Type: mstsc
3. Enter server IP: 203.0.113.50:3389
4. Click "Connect"

6. Accept certificate (first connection)
7. Desktop appears - full GUI access to server

(Opens RDP connection with 1920x1080 resolution)

# Method 3: PowerShell
PS> cmdkey /generic:203.0.113.50 /user:Administrator /pass:password

⚠️ RDP Security Considerations

  • Default Port: Port 3389 frequently scanned by attackers
  • Brute Force: RDP vulnerable to password guessing attacks
  • Latest Patches: RDP has had critical remote code execution vulnerabilities
  • Mitigation: Use VPN gateway, change RDP port, implement account lockout

Server Security Hardening Best Practices

Securing Publicly Deployed Infrastructure

Defense-in-Depth Approach

Multiple layers of security controls protect systems from various attack vectors:

  • Layer 1 (Network): Firewall, port filtering, rate limiting
  • Layer 2 (Service): SSH keys, disabled root, strong passwords
  • Layer 3 (Application): Input validation, secure coding
  • Layer 4 (Detection): Logging, monitoring, IDS/IPS

✓ Critical Security Hardening Checklist

  • FTP Server: Non-standard port, disable anonymous, chroot jail enabled, strong passwords
  • Web Server: Hide version info, disable directory listing, HTTP strict headers
  • SSH: Strong authentication (public keys), disable root login, non-standard port optional
  • Database: Change default credentials, least privilege permissions, backups
  • Logging: Comprehensive logging enabled, centralized log storage, log rotation
  • Firewall: Whitelist approach, restrict non-essential ports, geographic restrictions
  • Updates: Security patches applied promptly, automatic updates for critical services

Quick Hardening Commands

# Disable SSH root login
$ sudo nano /etc/ssh/sshd_config
  PermitRootLogin no
$ sudo systemctl restart ssh

# Change FTP port from 21 to 2121
$ sudo nano /etc/vsftpd.conf
  listen_port=2121
$ sudo systemctl restart vsftpd

# Hide Apache version
$ sudo nano /etc/apache2/conf-available/security.conf
  ServerTokens Prod
$ sudo systemctl restart apache2

# Set strict file permissions
$ sudo chmod 750 /var/www/html
$ sudo chown www-data:www-data /var/www/html

# Enable firewall
$ sudo ufw enable
$ sudo ufw default deny incoming
$ sudo ufw allow 22/tcp     # SSH
$ sudo ufw allow 80/tcp     # HTTP
$ sudo ufw allow 443/tcp    # HTTPS
$ sudo ufw allow 2121/tcp   # Custom FTP port

⚠️ Common Deployment Mistakes

  • Default Credentials: Using default passwords for root/admin accounts
  • Unpatched Systems: Running outdated software with known vulnerabilities
  • Open Ports: Leaving unnecessary services accessible to public
  • No Logging: Unable to detect attacks after they happen
  • Weak Encryption: Using HTTP instead of HTTPS for sensitive data
  • Information Disclosure: Server banners revealing version/OS information

Security Monitoring Workflow

  • 1. Continuous Logging: All services log to centralized location
  • 2. Regular Review: Examine logs for suspicious patterns
  • 3. Alert Triggers: Automated alerts for critical events
  • 4. Incident Response: Rapid action when threats detected
  • 5. Log Retention: Keep historical logs for forensic analysis
