🔧 The Domain Name System (DNS)

What is DNS?

The Domain Name System (DNS) is a critical internet service that translates human-readable domain names into machine-readable IP addresses. Without DNS, the internet would be unusable for average users.

Without DNS:
✗ Users memorize IP addresses
✗ Website access confusing
✗ Static IPs required
✗ Very user-unfriendly
With DNS:
✓ Easy-to-remember names
✓ Dynamic IP addresses
✓ Load balancing support
✓ User-friendly internet

DNS Server Hierarchy

DNS uses a hierarchical structure to distribute the load of name resolution:

Root Nameservers

• 13 root servers worldwide
• Direct to TLD servers
• Known to all recursive resolvers
• Published globally
• Never directly queried by users

TLD Nameservers

• Manage .com, .org, .in, etc.
• Contain NS records
• Direct to authoritative servers
• Maintained by registry operators
• Handle millions of domains

Authoritative Nameservers

• Hold actual DNS records
• Managed by domain owners
• Provide authoritative answers
• Return IP addresses directly
• Can be multiple servers

Recursive Resolvers

• Provided by ISPs
• Handle client requests
• Query other servers
• Cache results
• Improve performance

Why DNS is Critical

🚨 Single Point of Failure:
If DNS servers go down:
• Users cannot access websites by domain
• Only direct IP access possible (impractical)
• Email delivery fails
• Most online services become inaccessible
• Internet becomes effectively non-functional
95% Internet Dependency - Most critical service

⚙️ DNS Resolution Process

How DNS Resolution Works

When you type a domain into your browser, multiple DNS queries occur in a specific sequence:

1️⃣

User Request

User types domain name (e.g., gu.ac.in) into browser

2️⃣

Recursive Resolver Query

Browser sends request to recursive resolver (usually ISP's DNS like 8.8.8.8)

3️⃣

Root Nameserver Query

Resolver queries root nameserver asking: "Where is .in?"

4️⃣

TLD Server Query

Root responds with TLD server. Resolver queries: "Where is gu.ac.in?"

5️⃣

Authoritative Server Query

TLD responds with authoritative server. Resolver queries the actual IP

✓

Response & Resolution

Authoritative server returns IP address to resolver, then to browser

DNS Query Types

Recursive Query

What: Resolver must provide complete answer

Used by: Client to resolver

Response: IP address or error

Iterative Query

What: Server provides best answer it has

Used by: Resolver to root/TLD/auth servers

Response: Answer or referral to next server

Non-Recursive Query

What: Query to cached DNS result

Used by: Accessing cached data

Response: Cached answer without lookup

A Record (Address Record)

Purpose: Maps domain name to IPv4 address

Example: gu.ac.in → 10.0.0.1

Usage: Primary method for website access - most critical record

Created: Manually by domain administrator

TTL: Usually 3600 seconds (1 hour)

AAAA Record (IPv6 Address)

Purpose: Maps domain name to IPv6 address (128-bit)

Example: gu.ac.in → 2001:db8::1

Usage: For IPv6-enabled infrastructure (future internet)

Created: Manually by domain administrator

Trend: Becoming increasingly important as IPv4 depletes

MX Record (Mail Exchange)

Purpose: Directs email to correct mail servers

Example: user@gu.ac.in → mail.gu.ac.in (priority 10)

Usage: Critical for email delivery - must exist for email to work

Priority: Multiple MX records supported with priority levels

Created: Manually by domain administrator

NS Record (Nameserver)

Purpose: Indicates which DNS server is authoritative for domain

Example: ns1.godaddy.com, ns2.godaddy.com

Usage: Critical for DNS delegation to multiple servers

Created: Automatically provided by registrar

Modification: Requires careful coordination with registrar

CNAME Record (Canonical Name)

Purpose: Creates domain name aliases - one domain points to another

Example: www.gu.ac.in → gu.ac.in

Usage: Allows multiple domain names for same resource

Created: Manually by domain administrator

Limitation: Cannot have CNAME at zone apex (@)

TXT Record (Text Record)

Purpose: Stores arbitrary text information for verification and security

Common Uses:

  • SPF: Specifies authorized mail servers
  • DKIM: Email authentication key
  • DMARC: Email policy enforcement
  • Domain Verification: Google, Microsoft, other services

Created: Manually by domain administrator

Visibility: Publicly visible - use for public verification only

🔎 Passive Footprinting Techniques

What is Passive Footprinting?

Passive Footprinting: Gathering information about a target through publicly available sources WITHOUT direct system interaction. This is legal, ethical reconnaissance.

Footprinting Techniques

WHOIS Lookup

Information Retrieved:

  • Domain registrar (GoDaddy, Namecheap)
  • Registration date
  • Expiration date
  • Registrant info (if public)
  • Nameservers
  • Admin contact

DNS Enumeration

Reveal:

  • A records (IP addresses)
  • MX records (email servers)
  • NS records (nameservers)
  • TXT records (email auth)
  • CNAME records (aliases)
  • SOA records (zone info)

IP History

Tools: IP2Location, MaxMind

  • Previous IPs used
  • Geographic location
  • Cloud provider (AWS, Azure)
  • Data center location
  • Hosting provider

Email Footprinting

Information Gathered:

  • Employee email addresses
  • Company domain patterns
  • Email server location
  • Email security measures
  • SMTP configuration

Legal & Ethical Considerations

⚠️ CRITICAL LEGAL NOTE:
While passive footprinting uses ONLY public information, remember:
• Always get written authorization before testing
• Don't assume public = permission to test
• Document all authorization carefully
• Be prepared to explain your actions
• Follow responsible disclosure practices

🔍 Subdomain Enumeration

Understanding Subdomains

Subdomains are additional domain names under a main domain. Organizations often create subdomains for different services, departments, or purposes.

Examples: www.google.com, mail.google.com, calendar.google.com, drive.google.com

Why Subdomains Matter to Security

  • Focus Gaps: Organizations often focus security on main domain only
  • Weaker Security: Subdomains frequently have lower security standards
  • Service Diversity: Host different services (mail, admin, dev, staging, API)
  • Backdoors: Unsecured subdomains can become entryways for attackers
  • Vulnerability Variance: Different services expose different vulnerabilities

📧 Email Security & Footprinting

Email Authentication Standards

SPF (Sender Policy Framework)

Purpose: Prevents email spoofing

How: Domain publishes authorized sending servers in TXT record

Result: Receiving server verifies email came from authorized IP

Status Check: Pass/Fail

DKIM (DomainKeys Identified Mail)

Purpose: Digitally signs emails

How: Sender encrypts with private key, receiver verifies with public key

Result: Proves email authenticity and prevents tampering

Status Check: Pass/Fail

DMARC (Domain-based Message Authentication)

Purpose: Policy enforcement for SPF/DKIM

How: Domain publishes DMARC policy in TXT record

Policy Options: none, quarantine, reject

Result: Sends failure reports to domain owner

📚 Historical Data & Wayback Machine

What is the Wayback Machine?

The Wayback Machine (archive.org) stores historical snapshots of websites, allowing you to see how websites have evolved over time.

Website Archive: 25+ years of web history stored and publicly accessible. You can view any website from any year.

Value for Security Professionals

Historical Website Snapshots

  • Previous website designs
  • Changes over years/months
  • Added/removed content
  • Security improvements
  • Exposed administrator panels

Information Revealed

  • Former email addresses
  • Outdated security measures
  • Development servers
  • Admin contact info
  • API documentation

⚠️ DNS-Based Threats & Risks

Common DNS Attacks

DNS Spoofing

Attack: Attacker intercepts DNS queries and returns fake IP

Result: User redirected to attacker's server

Appearance: User thinks they're on legitimate website

Risk: Phishing, credential theft, malware

DNS Hijacking

Attack: Gain access to domain registrar

Action: Change nameservers to attacker's servers

Result: Complete control of domain

Risk: Email interception, website defacement

DNS Enumeration by Attackers

Goal: Find unsecured subdomains and services

Discover: Backup systems, test servers, admin panels

Identify: Mail servers, development infrastructure

Risk: Backdoor access, sensitive data exposure

DNS Amplification DDoS

Attack: Use DNS servers to amplify DDoS traffic

Method: Send small request, receive large response

Result: Overwhelming bandwidth consumption

Risk: Service unavailability

Information Leakage Risks

What Can Be Exposed

  • Email Addresses: From WHOIS, subdomains, historical data
  • Employee Data: Found in old website snapshots, email servers
  • Infrastructure Details: Server locations, cloud providers, ISPs
  • Security Gaps: Outdated systems, unpatched vulnerabilities
  • Business Information: Structure, departments, projects, partnerships
  • Sensitive URLs: Admin panels, API endpoints, development sites
75% Risk Level - Significant organizational impact

🛡️ Protection & Prevention Methods

Securing DNS Infrastructure

DNSSEC (DNS Security Extensions)

Purpose: Cryptographically sign DNS records

How: Exchange signed records using public key cryptography

Protection: Prevents DNS spoofing and cache poisoning

Benefit: Ensures data integrity and authenticity

DNS over HTTPS (DoH)

Purpose: Encrypt DNS queries in transit

How: DNS requests sent over HTTPS instead of plain DNS

Protection: Privacy from ISPs and network observers

Benefit: Prevents DNS manipulation on networks

Protecting Against Footprinting

  • WHOIS Privacy: Hide registrant information from public records
  • Regular Audits: Review WHOIS, DNS records for exposure
  • Monitor Archive.org: Check for sensitive data in old snapshots
  • Email Protection: Don't expose employee emails to public
  • Subdomain Inventory: Maintain complete list of all subdomains
  • Security Assessment: Apply equal security to all subdomains
  • Remove Unused: Delete subdomains no longer needed
  • Monitor Rogue: Detect unauthorized subdomain creation

Incident Response

If DNS is Compromised:

  1. Immediately change all DNS credentials
  2. Verify all DNS records are legitimate
  3. Check with registrar for unauthorized changes
  4. Monitor for malicious traffic patterns
  5. Update nameservers if necessary
  6. Notify users of security incident
  7. File incident report with registrar
  8. Implement additional monitoring

⚖️ Ethical Considerations & Legal Boundaries

Legal vs Illegal Reconnaissance

✓ Legal Passive Techniques

  • WHOIS lookups on any domain
  • DNS record queries
  • Archive.org historical data
  • Publicly available information
  • Search engine queries
  • Published news articles

Why Legal: All public info, no system access

✗ Illegal Activities

  • Unauthorized reconnaissance
  • Port scanning without permission
  • Vulnerability testing without consent
  • Accessing non-public data
  • Network probing
  • Exploitation attempts

Consequences: Federal charges, imprisonment, fines

Ethical Hacker Responsibilities

Always Remember

  • Only conduct reconnaissance on authorized targets
  • Get written authorization BEFORE testing
  • Use proven ethical hacking methodologies
  • Respect privacy and confidentiality
  • Report findings responsibly
  • Follow responsible disclosure guidelines
  • Maintain professional standards
  • Document all authorization carefully

Authorization Requirements

Requirement Details Importance

Defined Scope Clear target systems and methods Critical
Testing Window Specific dates and times allowed Important
Contact Info Escalation path and stakeholder contact Important
Documentation Keep copies of all authorization Critical
