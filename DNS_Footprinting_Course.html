🔧 The Domain Name System (DNS)

What is DNS?

DNS (Domain Name System) is a critical internet service that translates human-readable domain names (like google.com) into machine-readable IP addresses (like 172.217.2.46). Without DNS, users would need to memorize IP addresses to access websites.

Why DNS is Essential

Without DNS:

  • Users would need to memorize numerical IP addresses
  • Website access would be confusing and impractical
  • Internet would be much less user-friendly
  • Websites would need unique, static IP addresses

With DNS:

  • Users access websites by easy-to-remember names
  • IP addresses can change without affecting users
  • Multiple servers can share the same domain
  • Enables flexible server management

How DNS Works: The Resolution Process

Step-by-Step Process:
  1. User Request: User types domain (e.g., gu.ac.in) in browser
  2. Client DNS Query: Client sends request to recursive resolver (usually ISP's DNS)
  3. Root DNS Query: Resolver queries root nameserver
  4. TLD Query: Root directs to Top Level Domain (.in, .com) server
  5. Authoritative Query: TLD directs to authoritative nameserver
  6. IP Resolution: Authoritative server returns the IP address
  7. Connection Established: Browser connects to web server using IP
  8. Website Loading: Server responds and website loads

DNS Server Roles

Recursive Resolver
• Usually provided by ISP
• Handles client requests
• Queries other servers
• Caches results for performance
Root Nameserver
• Points to TLD servers
• Part of DNS hierarchy
• Known to all resolvers
• Distributed globally
TLD Nameserver
• Manages top-level domains
• Handles .com, .org, .in, etc.
• Directs to authoritative servers
• Maintained by registry operators
Authoritative Nameserver
• Holds actual DNS records
• Managed by domain owner
• Provides authoritative answers
• Returns IP addresses directly

Critical DNS Dependency

⚠️ DNS Outage Impact:
If DNS servers are down or not responding:
• Users cannot access websites by domain name
• Only way to access is by direct IP address (impractical)
• Email delivery may fail
• Many online services become inaccessible
DNS is a critical single point of failure for the internet

📋 DNS Resource Records

DNS servers maintain databases called Resource Records that contain mappings between domain names and various information.

Major DNS Record Types

A Record (Address Record)

Purpose: Maps domain name to IPv4 address
Example: gu.ac.in → 10.0.0.1
Usage: Primary method for website access
Created: Manually by domain administrator

AAAA Record

Purpose: Maps domain name to IPv6 address
Example: gu.ac.in → 2001:db8::1
Usage: For IPv6-enabled infrastructure
Created: Manually by domain administrator

NS Record (Nameserver Record)

Purpose: Indicates which DNS server is authoritative
Example: ns1.godaddy.com is responsible for gu.ac.in
Usage: Critical for DNS delegation
Created: Automatically provided by registrar

MX Record (Mail Exchange Record)

Purpose: Directs email to correct mail servers
Example: Email for user@gu.ac.in → mail.gu.ac.in
Usage: Required for email delivery
Created: Manually by domain administrator

CNAME Record (Canonical Name)

Purpose: Creates domain name aliases
Example: www.gu.ac.in → gu.ac.in
Usage: Allows multiple domain names for same resource
Created: Manually by domain administrator

TXT Record

Purpose: Stores text information
Usage: SPF, DKIM, DMARC verification
Example: "v=spf1 ip4:192.168.0.1 ~all"
Created: Manually by domain administrator

SOA Record (Start of Authority)

Purpose: Contains DNS zone information
Contains: Primary nameserver, admin email, serial number
Usage: Zone management and synchronization
Created: Automatically by DNS system

Record Management

Record Type Purpose Created By Critical?
A / AAAA IP resolution Manual ✓ Yes
NS DNS delegation Automatic ✓ Yes
MX Email routing Manual ✓ Yes
CNAME Aliases Manual ○ No
TXT Security verification Manual ✓ Yes
SOA Zone information Automatic ✓ Yes

🔍 Subdomain Enumeration

Subdomains are additional domain names under a main domain. For example, www.google.com, mail.google.com, and calendar.google.com are all subdomains of google.com.

What are Subdomains?

Subdomain: A domain that is part of a larger domain. Format: [subdomain].[domain].[extension]
Example: www.gu.ac.in where "www" is the subdomain of "gu.ac.in"

Why Subdomains Matter

From a Security Perspective:

  • Organizations often focus security on main domain only
  • Subdomains frequently have weaker security
  • Subdomains often host different services (mail, admin, dev, etc.)
  • Unsecured subdomains can be backdoors for attackers
  • Can expose different services and vulnerabilities

Subdomain Enumeration Process

Subdomain Finder Approach:
1. Enter target domain (e.g., gu.ac.in)
2. Tool queries DNS records for subdomains
3. Receives list: admin.gu.ac.in, mail.gu.ac.in, etc.
4. Shows IP addresses for each subdomain
5. Reveals services running on each subdomain

Tools for Subdomain Discovery

Online Tools

  • Subdomain Finder (c99.nl)
  • DNS.mxtoolbox.com
  • ViewDNS.net
  • MXToolbox.com
  • BuiltWith.com

Command Line Tools

  • nslookup - Query DNS records
  • dig - DNS lookup utility
  • host - DNS lookup command
  • whois - Domain information
  • dnsrecon - DNS scanning tool

Information Revealed

Each discovered subdomain reveals:

  • Domain Name: The subdomain itself
  • IP Address: Server location (often public IPs)
  • Service Type: Web server, mail server, admin panel, etc.
  • Organization Department: Which team operates it
  • Geographic Location: Where server is hosted (if cloud)
  • Security Status: Whether it's properly secured

🔎 Passive Footprinting Techniques

Passive footprinting involves gathering information about a target without directly interacting with their systems. This is legal and ethical reconnaissance.

What is Passive Footprinting?

Passive Footprinting: The process of gathering information about a target organization through publicly available sources without direct system interaction. This is non-intrusive and legal.

Common Passive Techniques

1. WHOIS Lookup

Purpose: Get domain registration information
Information Retrieved:
• Domain registrar (GoDaddy, Namecheap, etc.)
• Registration date
• Expiration date
• Registrant information
• Nameservers
• Admin contact

2. DNS Record Lookup

Reveal:
• A records (IP addresses)
• MX records (email servers)
• NS records (nameservers)
• TXT records (email authentication)
• CNAME records (aliases)
• SOA records (zone information)

3. IP History & Geolocation

Tools: IP2Location, MaxMind, GeoIP
Information:
• Previous IP addresses used
• Geographic location of servers
• Cloud provider (AWS, Azure, GCP)
• Data center location
• Hosting provider information
⚠️ IMPORTANT LEGAL NOTE:
While passive footprinting is legal, always remember:
• Only perform reconnaissance on authorized targets
• If targeting unfamiliar organizations, you may face legal consequences
• Always have written permission before security testing
• Keep documentation of authorization
• Be prepared to explain why you conducted reconnaissance

📧 Email Security & Footprinting

Understanding Email Headers

Email headers contain metadata that reveals how the email traveled through the internet and whether it's legitimate.

Email Authentication Standards

SPF (Sender Policy Framework)

Purpose: Prevents email spoofing by specifying authorized sending servers

How It Works:

  • Domain publishes list of authorized mail servers in TXT record
  • Receiving server checks SPF record
  • Verifies email came from authorized IP

Status: Pass/Fail

DKIM (DomainKeys Identified Mail)

Purpose: Digitally signs emails to prove authenticity

How It Works:

  • Sender encrypts email with private key
  • Receiver verifies with public key (in TXT record)
  • Proves email wasn't tampered with

Status: Pass/Fail

DMARC (Domain-based Message Authentication)

Purpose: Policy enforcement for SPF/DKIM

How It Works:

  • Domain publishes DMARC policy in TXT record
  • Specifies what to do if SPF/DKIM fail
  • Sends failure reports to domain owner

Policy: none, quarantine, reject

Identifying Suspicious Emails

Red Flags for Phishing/Spam:
• SPF: FAIL
• DKIM: FAIL
• DMARC: FAIL
• Domain name slightly different from legitimate
• Sender IP doesn't match domain location
• Generic greeting (Hi user vs Hi John)
• Suspicious links or attachments
• Urgency or threats to click

Checking if Your Email was Hacked

Tools: Have I Been Pwned (haveibeenpwned.com)

Information Revealed:

  • Whether your email is in known breaches
  • Which services were compromised
  • What data was exposed (if available)
  • When the breach occurred

If Compromised:

  • Change password immediately
  • Enable 2FA/MFA on that account
  • Change passwords for linked services
  • Monitor account for suspicious activity

📚 Historical Data & Wayback Machine

The Wayback Machine (archive.org) stores historical snapshots of websites, allowing you to see how websites have changed over time.

What Can the Wayback Machine Show?

Historical Website Snapshots

  • Previous website designs
  • Changes over years/months
  • Added/removed content
  • Security improvements
  • Service changes

Activity Timeline

  • When site was first deployed
  • Frequency of updates
  • Development patterns
  • Periods of inactivity
  • Recent changes

Using Wayback Machine for Reconnaissance

Process:
1. Visit archive.org
2. Enter target domain
3. See timeline of captures
4. Click specific date to view that snapshot
5. Compare snapshots from different time periods
6. Note security changes, new services, vulnerabilities

Value for Penetration Testers

Information Revealed:

  • Former email addresses (might work for password reset)
  • Outdated security measures (historical vulnerabilities)
  • Development servers that were previously exposed
  • Administrator contact information
  • API endpoints or documentation
  • Subdomain history

Real-world Case Study

Scenario: Verifying legitimate organizations

A friend received a recruitment email from a company offering a high-paying position (1 crore per hour). The email looked professional, but had red flags:
• Domain name was slightly different
• Used WHOIS lookup to check registration date
• Found domain was only registered 2 months prior
• Legitimate company should have much older domain
• Subsequent emails asked for visa processing fees
Conclusion: Obvious scam. This reconnaissance was crucial in preventing a fraud.

⚠️ Threats & Risks in DNS

DNS-Based Attacks

DNS Spoofing

How It Works:
• Attacker intercepts DNS queries
• Returns fake IP address
• User redirected to attacker's server
• User thinks they're on legitimate site
Risk: Phishing, credential theft, malware distribution

DNS Hijacking

How It Works:
• Attacker gains access to domain registrar
• Changes nameservers
• Points domain to attacker's server
• Users see attacker's fake website
Risk: Complete control of domain, email interception

DNS Enumeration by Attackers

Why Attackers Do It:
• Find unsecured subdomains
• Discover backup systems
• Identify test/development servers
• Find mail servers
• Locate admin panels
Risk: Backdoor access, sensitive data exposure

Information Leakage Risks

What Information Can Be Exposed:

  • Email Addresses: From WHOIS, subdomains, historical data
  • Employee Data: Found in old website snapshots, email servers
  • Infrastructure Details: Server locations, cloud providers
  • Security Gaps: Outdated systems, unpatched vulnerabilities
  • Business Information: Structure, departments, projects

🛡️ Protection & Prevention Methods

Securing DNS

DNSSEC (DNS Security Extensions)

Purpose: Cryptographically sign DNS records
How It Works:
• Signs DNS records with private key
• Receivers verify with public key
• Prevents DNS spoofing
• Ensures data integrity

Email Security

Implement SPF, DKIM, DMARC:
• Prevents email spoofing
• Authenticates senders
• Protects reputation
• Reduces phishing success rate

Subdomain Management

Best Practices:
• Maintain inventory of all subdomains
• Apply security to all subdomains equally
• Remove unused subdomains
• Monitor for rogue subdomains
• Regular security assessments

Footprinting Defense

Information Minimization

  • WHOIS Privacy: Hide registrant info from public records
  • Public Records: Review WHOIS, DNS records for exposure
  • Website Archive: Monitor archive.org for old content
  • Email Protection: Don't expose employee emails publicly

Monitoring

  • Regular DNS audits
  • Monitor for unauthorized subdomains
  • Check blacklist status regularly
  • Email security monitoring
  • Brand monitoring for domain squatting

Incident Response

If Compromised:
• Immediately change all DNS credentials
• Verify all DNS records are legitimate
• Check with registrar for unauthorized changes
• Monitor for malicious traffic
• Update nameservers if necessary
• Notify users of incident

⚖️ Ethical Considerations & Legal Boundaries

Legal vs Illegal Reconnaissance

✓ Legal Passive Footprinting

  • WHOIS lookups on any domain
  • DNS record queries
  • Archive.org historical data
  • Publicly available information
  • Search engine queries
  • Published news articles

Why Legal: All information is publicly available and no system access occurs

✗ Illegal Activities

  • Reconnaissance without authorization
  • Port scanning without permission
  • Vulnerability testing without consent
  • Accessing non-public data
  • Network probing
  • Attempting to exploit systems

Consequences: Federal charges, imprisonment, fines

Ethical Hacker Responsibilities

Always Remember:
• Only conduct reconnaissance on YOUR systems or authorized targets
• Get written authorization BEFORE testing
• Use only proven ethical hacking methodologies
• Respect privacy and confidentiality
• Report findings responsibly
• Follow disclosure guidelines
• Maintain professional standards
• Keep good documentation of authorization

Authorization Requirements

Before Any Security Testing:

  • Get written test authorization
  • Define scope clearly
  • Establish testing window (dates/times)
  • Document timeline
  • Identify authorized tester
  • Know escalation procedures
  • Have stakeholder contact information
