📡 Network Scanning Fundamentals

What is Network Scanning?

Network scanning is the process of submitting requests to a network and analyzing responses to identify active devices, their services, and potential vulnerabilities.

1️⃣

Network Assessment

Identify target network range and architecture

2️⃣

Host Discovery

Find which IPs are active and responding to requests

3️⃣

Port Scanning

Test each host for open ports and running services

4️⃣

Service Identification

Determine service versions and operating systems

5️⃣

Vulnerability Detection

Identify potential exploitable weaknesses

Key Network Scanning Concepts

IP Addresses & Devices

  • Each device on network has unique IP
  • Class C network: 192.168.1.0 - 192.168.1.255
  • 254 usable host addresses per network
  • Scanner queries all possible IPs
  • Only responding devices are "live hosts"

Port Numbers

  • Services run on specific port numbers
  • Port range: 0-65535
  • Well-known ports: 0-1023 (HTTP:80, SSH:22)
  • Registered ports: 1024-49151
  • Dynamic/Private: 49152-65535

Common Port Numbers

  • 21 - FTP (File Transfer)
  • 22 - SSH (Secure Shell)
  • 80 - HTTP (Web Browsing)
  • 443 - HTTPS (Secure Web)
  • 3306 - MySQL (Database)

Port States

  • Open: Service actively accepting
  • Closed: Port not listening
  • Filtered: Firewall blocking
  • Unfiltered: Port accessible, state unknown
  • Open|Filtered: Can't determine state

📡 ICMP Protocol & Ping Operations

What is ICMP?

ICMP (Internet Control Message Protocol): A network layer protocol used by network devices to diagnose network communication issues and test host connectivity. Works with the OSI model's network layer (Layer 3).

Common ICMP Messages:

Echo Request (Ping)

Type 8 - "Are you alive?"

Sent by source to destination

Seeks connectivity confirmation

Echo Reply

Type 0 - "Yes, I'm alive!"

Sent in response to echo request

Confirms device is reachable

Destination Unreachable

Type 3 - "Can't reach that destination"

Network problems or blocking

Helps diagnose routing issues

Time Exceeded

Type 11 - "TTL expired"

Packet took too many hops

Used by traceroute tool

How Ping Works (Step by Step)

Ping Request/Reply Sequence

PC1 (Source: 192.168.1.1) → PC2 (Destination: 192.168.1.10)

Step 1: User runs command

$ ping 192.168.1.10

Step 2: PC1 creates ICMP Echo Request packet

  • Type: Echo Request (8)
  • Source IP: 192.168.1.1
  • Destination IP: 192.168.1.10
  • Source MAC: PC1's MAC address
  • Optional: Data payload

Step 3: PC2 receives Echo Request

  • PC2 recognizes as "someone checking if alive"
  • PC2 validates packet integrity
  • PC2 prepares response

Step 4: PC2 sends Echo Reply packet

  • Type: Echo Reply (0)
  • Source IP: 192.168.1.10 (echoing PC2's IP)
  • Destination IP: 192.168.1.1 (echoing PC1's IP)
  • Returns same data payload

Step 5: PC1 receives Echo Reply

  • PC1 calculates response time (RTT)
  • PC1 verifies received data matches sent
  • PC1 displays success to user

Reply from 192.168.1.10: bytes=32 time=2ms TTL=64

Ping Output Analysis

$ ping 8.8.8.8 -c 4

64 bytes from 8.8.8.8: icmp_seq=2 ttl=119 time=24.8 ms
64 bytes from 8.8.8.8: icmp_seq=3 ttl=119 time=25.1 ms
64 bytes from 8.8.8.8: icmp_seq=4 ttl=119 time=24.9 ms

--- 8.8.8.8 statistics ---

Field Meaning Value Example

🔌 Ports, Services & Protocols

Understanding Protocols

Protocol: A set of rules defining how two devices communicate with each other. Each protocol has specific responsibilities and operates on defined port numbers.

How Protocols Work:

  1. Web Browsing: You type URL → HTTP/HTTPS Protocol → Port 80/443
  2. Remote Access: You SSH → SSH Protocol → Port 22
  3. File Transfer: You FTP → FTP Protocol → Port 21
  4. Email Sending: Email App → SMTP Protocol → Port 25/587
  5. Email Receiving: Email App → IMAP Protocol → Port 143/993

Common Protocols & Ports

Port Protocol Service Connection Type Security
20-21 FTP File Transfer TCP Unencrypted (Legacy)
22 SSH Secure Shell TCP Encrypted, Recommended
25 SMTP Email Sending TCP Can be encrypted
53 DNS Domain Resolution UDP/TCP Plain text
80 HTTP Web Browsing TCP Unencrypted
110 POP3 Email Reception TCP Can be encrypted
143 IMAP Email Reception TCP Can be encrypted
443 HTTPS Secure Web TCP Encrypted
3306 MySQL Database TCP Can be encrypted
3389 RDP Remote Desktop TCP Can be encrypted

Port States Explained

🟢 OPEN Port

Meaning: Service actively listening

Response: Accepts connections

Threat Level: Potential vulnerability if unpatched

Example: Web server on port 80

🔴 CLOSED Port

Meaning: No service listening

Response: Sends RST packet back

Threat Level: Lower, but host confirmed alive

Example: Unused port number

🟡 FILTERED Port

Meaning: Firewall blocking

Response: No response or dropped packets

Threat Level: Protected by firewall

Example: Corporate firewall blocking inbound

⚪ UNFILTERED Port

Meaning: Accessible but state unknown

Response: Responds to packets

Threat Level: Moderate uncertainty

Example: Some firewall configurations

NMAP (Network Mapper): The industry-standard open-source tool for network discovery, port mapping, and security auditing. Used by security professionals worldwide for legitimate penetration testing and network administration.

Key Capabilities:

Host Discovery

  • Identify live hosts on network
  • Determine which IPs are active
  • Ping scan capabilities

Port Scanning

  • Scan all 65,535 ports
  • Determine open/closed status
  • Multiple scanning techniques
  • Firewall evasion

Service Detection

  • Identify service versions
  • Determine software names
  • Detect running services
  • Version fingerprinting

Vulnerability Detection

  • Identify known vulnerabilities
  • Script-based testing
  • Firewall configuration issues
  • Security weakness discovery

CIDR Notation

CIDR (Classless Inter-Domain Routing): A compact way to specify IP address ranges by indicating the number of bits that are "on" in the network mask. Format: IP/bits

CIDR Examples:

CIDR Notation Subnet Mask Network Range Total Hosts Usable Hosts
192.168.1.0/24 255.255.255.0 192.168.1.0 - 192.168.1.255 256 254
192.168.0.0/23 255.255.254.0 192.168.0.0 - 192.168.1.255 512 510
192.168.0.0/22 255.255.252.0 192.168.0.0 - 192.168.3.255 1024 1022
10.0.0.0/8 255.0.0.0 10.0.0.0 - 10.255.255.255 16.7M 16.7M-2

Why CIDR Notation?

  • Shorter: "192.168.1.0/24" vs "192.168.1.0 with subnet mask 255.255.255.0"
  • Easier: Number of bits = network size
  • Flexible: Can specify any network size
  • Standard: Used across all networking tools

TCP SYN Scan (Stealth Scan)

Command: nmap -sS 192.168.1.10

How It Works:

  1. Attacker sends TCP SYN packet (connection request)
  2. If port OPEN: Target responds with SYN-ACK
  3. Attacker sends RST (reset) instead of completing connection
  4. Connection never fully established (harder to log)

Advantages: Stealthy, faster, doesn't complete connections

Disadvantages: Requires root/admin privs, some firewalls detect anyway

TCP Connect Scan

Command: nmap -sT 192.168.1.10

How It Works:

  1. Attacker sends SYN packet
  2. Target responds with SYN-ACK (if port open)
  3. Attacker sends ACK (completes three-way handshake)
  4. Full TCP connection established
  5. Attacker then closes connection properly

Advantages: Works without special privileges, reliable

Disadvantages: Leaves full logs, easily detected, slower

FIN Scan

Command: nmap -sF 192.168.1.10

How It Works:

  1. Sends TCP packet with FIN flag set (connection termination)
  2. If port OPEN: No response (RFC compliant)
  3. If port CLOSED: Target sends RST-ACK packet
  4. Uses this asymmetry to determine port state

Logic: Assumes open ports ignore FIN packets to non-existent connections

Limitation: Windows systems don't follow RFC, may not work

Detectability: Shows "filtered" on many networks due to firewalls

NULL Scan

Command: nmap -sN 192.168.1.10

How It Works:

  1. Sends TCP packet with NO flags set
  2. If port OPEN: No response
  3. If port CLOSED: Target sends RST packet
  4. Differentiates based on responses

Similar to: FIN scan, Xmas scan (all RFC variations)

Problem: Windows doesn't respond correctly, results may be unreliable

Xmas Scan

Command: nmap -sX 192.168.1.10

How It Works:

  1. Sends TCP packet with FIN, PSH, URG flags all set
  2. Called "Xmas" because flags light up like Christmas lights
  3. Same logic as FIN/NULL: looks for asymmetrical responses
  4. If port OPEN: No response
  5. If port CLOSED: RST-ACK response

Note: "Xmas tree" is the nickname for this technique

Reliability: Limited, Windows systems don't work, firewalls often filter

UDP Scan

Command: nmap -sU 192.168.1.10

How It Works:

  1. Sends UDP packets to target ports
  2. If port OPEN: May respond with data or no response
  3. If port CLOSED: Target sends ICMP port unreachable
  4. If ICMP blocked: Shows as "open|filtered"

Services: DNS (53), SNMP (161), NTP (123), DHCP (67)

Challenge: UDP is connectionless, harder to determine state

Speed: Much slower than TCP scanning due to ICMP rate limiting

🔀 TCP vs UDP Protocols

Fundamental Differences

Aspect TCP UDP
Connection Type Connection-Oriented Connectionless
Connection Setup Three-way handshake (SYN-SYN/ACK-ACK) No handshake, sends data directly
Reliability Reliable (100% delivery guarantee) Unreliable (no delivery guarantee)
Error Handling Error detection & correction No error correction
Speed Slower (due to handshake & verification) Faster (no verification overhead)
Ordering Packets arrive in order Packets may arrive out of order
Flow Control Yes (prevents overwhelming receiver) No (sender sends at own pace)
Header Size 20 bytes (larger) 8 bytes (smaller)
Common Uses Email, Web, File Transfer, SSH DNS, Video Streaming, Online Games, VoIP
Port Examples 80(HTTP), 443(HTTPS), 22(SSH), 25(SMTP) 53(DNS), 123(NTP), 161(SNMP), 67(DHCP)

Three-Way Handshake (TCP)

TCP Connection Establishment

Client (Source) → Server (Destination)

Step 1 - SYN: Client sends synchronization packet "I want to connect"

Flag: SYN | Sequence: 1000 | Data: Let's connect

Step 2 - SYN-ACK: Server acknowledges and responds "I'm ready to connect"

Flag: SYN-ACK | Sequence: 2000 | Acknowledgment: 1001

Step 3 - ACK: Client acknowledges "Great, connection established"

Flag: ACK | Sequence: 1001 | Acknowledgment: 2001

Result: Connection established, data can now flow

Why This Matters for Scanning

  • TCP SYN Scan: Stops at step 2, never completes handshake (stealthy)
  • TCP Connect Scan: Completes all 3 steps (full connection)
  • Firewall Logging: Different logging for partial vs complete connections
  • Detection Evasion: SYN scan is less obvious than connect scan

UDP: No Handshake, Direct Sending

UDP Connection Process

Client (Source) → Server (Destination)

Step 1: Client immediately sends data packet

No setup, no handshake, data goes immediately

Step 2: Server processes (if there is one listening) or silently drops

No acknowledgment, no verification

Result: Fast but unreliable - no guarantee data arrived

Analogy: UDP is like shouting in a crowded room. You don't wait to confirm people heard you, you just shout.

When TCP is Used vs UDP

✓ Use TCP When

  • All data must be delivered
  • Data must be in order
  • Retransmission is acceptable
  • Examples: Email, file downloads, SSH

✓ Use UDP When

  • Speed is critical
  • Occasional loss acceptable
  • Real-time is important
  • Examples: Streaming, gaming, VOIP

📦 Packet Capture & Analysis (Wireshark)

Understanding Packet Capture

Packet capture is the process of intercepting and logging network traffic for analysis. Every communication on a network travels as packets that can be captured and examined.

What Gets Captured?

  • All network traffic from/to your machine
  • Traffic between other devices on same network segment
  • Web browsing, email, SSH, DNS queries, everything
  • Can include sensitive data (passwords, credentials)
  • Packet capture is powerful reconnaissance tool

⚠️ Legal Note

Capturing traffic on networks you don't own/manage is illegal without authorization. Only capture on:

  • Your own machines
  • Networks you have written permission to test
  • Lab/test environments

Wireshark: Packet Capture Tool

What is Wireshark?

Industry-standard packet capture and analysis tool. Captures all network traffic passing through your network interface and displays it in readable format.

View in Wireshark Shows:

  • Source IP: Where packet came from
  • Destination IP: Where packet is going
  • Protocol: What type of packet (TCP, UDP, ICMP, DNS, etc.)
  • Port Numbers: Source and destination ports
  • Flags: TCP flags set (SYN, ACK, FIN, RST, etc.)
  • Data: Actual packet content/payload
  • TTL: Time To Live / hop count

🔥 Firewall Detection & Evasion

How Firewalls Block Scans

Firewall Detection Methods

  • Port Blocking: Drops/rejects packets to specific ports
  • Threshold Monitoring: Detects many ports hit in short time
  • Pattern Recognition: Recognizes NMAP signature patterns
  • IDS/IPS: Intrusion Detection/Prevention Systems
  • Rate Limiting: Blocks excessive traffic from one source
  • Stealth Techniques: Detects SYN scan, FIN scan attempts

Firewall Evasion Techniques

1. SYN Scan (-sS)

Type: Stealth/Half-Open scan

Advantage: Never completes connections, less logging

Disadvantage: Modern firewalls detect anyway

Detection: Requires root/admin privileges

2. FIN/NULL/Xmas Scans

Type: RFC-based evasion

Logic: Exploit RFC interpretation differences

Problem: Windows ignores RFC rules

Result: Unreliable but may bypass some firewalls

3. Fragmentation (-f)

Method: Break packets into fragments

Purpose: Bypass packet inspection

Command: nmap -f 192.168.1.10

Modern Impact: Most modern firewalls reassemble

4. Idle Scan

Method: Use zombie host to scan

Purpose: Hide actual source IP

Requirement: Find idle IP to use

Detectability: Still creates suspicious patterns

5. Slow Scans (-T0 to -T5)

Command: nmap -T0 192.168.1.10

Purpose: Avoid triggering rate thresholds

Trade-off: Much slower scanning

Effectiveness: Can evade automated detection

6. Spoofed Source IP

Method: Use fake source IP

Purpose: Hide true identity

Limitation: Responses won't reach attacker

Legal: Spoofing is illegal without authorization

🚨 Vulnerability Discovery Through Scanning

How Open Ports Lead to Vulnerabilities

Real Scenario: Vulnerable Web Server

Discovery Process:

  1. NMAP scan reveals port 80 open on 192.168.1.50
  2. Service detection shows: Apache 2.2.8 (OLD version)
  3. This Apache version has known buffer overflow
  4. Attacker finds matching exploit code
  5. Exploits vulnerability to gain shell access

Real Scenario: Exposed Database

Discovery & Exploitation:

  1. Scan finds port 3306 (MySQL) is OPEN
  2. Should NOT be open - database should be private!
  3. Attacker connects with default credentials
  4. Accesses entire customer database
  5. Exfiltrates sensitive data

🚨 NMAP Detection & IDS/IPS

How to Detect NMAP Scans

Signature-Based Detection

  • NMAP Signatures: Firewalls recognize NMAP-specific patterns
  • Packet Patterns: Unique TTL values, flags combinations
  • Timing Patterns: Regular intervals between packets
  • Behavior Analysis: Port scanning behavior is detectable

Anomaly-Based Detection

  • High Volume: Many packets in short time
  • Sequential Ports: Testing 1, 2, 3, 4... is suspicious
  • Connection Attempts: Many connections from single IP
  • Failed Connections: Lots of rejected attempts

IDS/IPS Systems

System Type Function Response Passive/Active
IDS Intrusion Detection Alerts/Logs only Passive (monitoring)
IPS Intrusion Prevention Blocks/Drops traffic Active (blocking)
Firewall Traffic Control Accept/Reject/Drop Active (enforcing rules)
WAF Web Application Protection Blocks malicious requests Active (web-focused)

Logs: Evidence of Scanning

What Gets Logged?

  • Firewall Logs: Dropped packets, connection attempts
  • IDS/IPS Logs: Suspicious activities flagged
  • Server Logs: Connection attempts, which ports accessed
  • Authentication Logs: Failed login attempts
  • Network Logs: Source IP, destination IP, port, timestamp
