Domain Controllers & Active Directory

Understanding centralized authentication and user management

What is a Domain Controller?

Domain Controller (DC): Server that manages user authentication and policy enforcement in a domain

  • Operating System: Windows Server (2016, 2019, 2022) or Linux/SAMBA
  • Primary Role: Authenticate users and enforce group policies
  • Centralized Database: SAM file equivalent for entire organization
  • Authorization: Determines what users can access and do
  • Replication: Multiple DCs replicate data for redundancy

Individual Machine vs Domain Environment

Aspect Standalone Machine Domain Environment
User Storage Local SAM file on each machine Centralized Active Directory
Authentication Local computer validates credentials Domain controller authenticates
Scalability 100+ computers = 100+ management points 100+ computers = 1 management point
Policy Enforcement Manual per-machine configuration Group Policy applied automatically
User Experience Login to specific computer only Login from any domain-joined computer
Maintenance Administrative nightmare at scale Centralized, efficient management

Domain Login Process

Step 1: User Initiation ▼

User presses Ctrl+Alt+Delete at login screen

Enters username and password for domain

Example: DOMAIN\username

Step 2: Domain Discovery ▼

Client machine locates domain controller via LDAP or DNS

Sends authentication request to DC

Step 3: Credential Validation ▼

Domain controller checks Active Directory database

Verifies username exists and password hash matches

Checks if user account is active (not disabled/locked)

Step 4: Kerberos Token Generation ▼

DC generates Kerberos TGT (Ticket Granting Ticket)

TGT encrypted with user's password hash

User receives Kerberos session ticket

Step 5: Access Granted ▼

User logged into desktop with domain credentials

Group policies applied from domain controller

User has access to domain resources based on group membership

Group Policy Application

Group Policy Object (GPO): Configuration template applied to users/machines

  • Password Policies: Complexity, age, history requirements
  • Desktop Restrictions: Disable wallpaper changes, screen saver settings
  • Application Control: Whitelist allowed applications, prevent installations
  • Security Policies: Audit settings, firewall rules, encryption requirements
  • Automatic Updates: Force Windows Update patches on schedule

Active Directory Architecture & Security Implications

Understanding AD structure and attack surface

Active Directory Components

  • Forest: Top-level organizational unit (usually one per organization)
  • Domain: Security boundary within forest (user auth, policy enforcement)
  • Organizational Units (OUs): Containers for users/computers with specific policies
  • Global Catalog: Searchable index of all AD objects
  • Schema: Defines object types and attributes
  • Replication: Changes replicated between domain controllers
🔴 AD ATTACK SURFACE:
  • Kerberos Attacks: TGT hijacking, pass-the-ticket, credential stuffing
  • LDAP Enumeration: Extract user list, group memberships, admin accounts
  • Domain Admin Account: If compromised → entire forest compromised
  • Unpatched Servers: Zero-day exploits on legacy domain controllers
  • GPO Abuse: Weaponized group policies distribute malware domain-wide
  • Golden Ticket: Forged Kerberos ticket with Domain Admin privileges

Active Directory Security Hardening

User & Account Security

  • Enforce strong passwords (20+ characters, complexity enabled)
  • Multi-factor authentication (MFA) for all user accounts
  • Limit administrative account usage (no daily use with admin privs)
  • Implement Just-In-Time (JIT) admin access
  • Monitor domain admin account usage in real-time

Domain Controller Protection

  • Isolate DCs on separate network segment
  • Restrict DC access to authorized administrators only
  • Enable DC replication signing and sealing
  • Regularly back up AD database
  • Apply latest security patches immediately
  • Disable anonymous LDAP queries

Monitoring & Detection

  • Monitor Kerberos ticket requests for anomalies
  • Alert on failed authentication attempts (threshold: 5+)
  • Track privileged account usage and access
  • Enable AD audit logging (directory service access, account management)
  • Use SIEM to centralize and correlate AD events
  • Monitor for DCSync attacks (credential replication)

Steghide: Enterprise-Grade File Hiding

Advanced steganography tool for professional data concealment

Steghide Capabilities

  • Supported Formats: JPEG, BMP, WAV, AU (audio), FLAC
  • Encryption: Rijndael-128 encryption built-in (not just compression)
  • Passphrase Protection: Secures hidden data with password
  • Compression: Automatic compression reduces file size
  • Stealth: Changes imperceptible to human eye/ear
  • Capacity: Up to 20-30% of carrier file can be hidden

Steghide Command Syntax & Examples

# EMBEDDING SECRET DATA INTO IMAGE
steghide embed -cf cover.jpg -ef secret.txt -p "passphrase"
# -cf: Cover File (carrier/image)
# -ef: Embed File (hidden file)
# -p: Passphrase (encryption key)

# EXTRACTING HIDDEN DATA FROM IMAGE
steghide extract -sf stego.jpg -p "passphrase"
# -sf: Stego File (image with hidden data)
# Extracts secret.txt automatically

# GET INFORMATION ABOUT STEGO FILE
steghide info stego.jpg
# Shows capacity, estimated hidden data size

# EMBED WITHOUT EXTENSION (HIDE FILE TYPE)
steghide embed -cf photo.jpg -ef database.zip -p "secret123" -sf output.jpg

# FORCE OVERWRITE EXISTING OUTPUT
steghide embed -cf image.jpg -ef confidential.doc -p "pass" -f

# BATCH PROCESSING MULTIPLE FILES
for file in *.txt; do
  steghide embed -cf base_image.jpg -ef "$file" -p "password" -sf "${file}.stego.jpg"
done

Data Exfiltration with Steghide

Scenario: Stealing Database ▼

Objective: Steal 100MB customer database without detection

Challenge: Email attachment filter blocks files >50MB

Solution: Compress and hide in legitimate image

Step 1: Prepare Data ▼
zip -9 database_backup.zip /path/to/database # Result: 100MB → 35MB compressed
Step 2: Obtain Carrier Image ▼
wget https://www.example.com/vacation_photo.jpg # Download high-resolution image (60MB) # Steghide capacity: ~15-20MB (sufficient)
Step 3: Hide Data ▼
steghide embed -cf vacation_photo.jpg \ -ef database_backup.zip -p "C0mpl3xP@ssw0rd" \ -sf business_trip_photos.jpg
Step 4: Send Innocuously ▼

Email: "vacation photos from NYC trip"

Attachment: business_trip_photos.jpg (60MB)

Content filter analysis: Legitimate JPG image ✓

Step 5: Extract at Destination ▼
steghide extract -sf business_trip_photos.jpg \ -p "C0mpl3xP@ssw0rd" # Recovers: database_backup.zip unzip database_backup.zip # Full database restored successfully

Detection Challenges

  • No Visual Differences: Image appears identical to naked eye
  • File Size: Can appear normal (steghide adds minimal overhead)
  • Metadata Preserved: EXIF data remains unmodified
  • Custom Passphrase: Only sender/recipient knows encryption key
  • Advanced Detection: Statistical analysis required (chi-square test, entropy)

OutGuess: DCT-Based Steganography

Advanced JPEG steganography using DCT coefficient manipulation

OutGuess vs Steghide

Key Difference: OutGuess modifies JPEG DCT (Discrete Cosine Transform) coefficients directly

Aspect Steghide OutGuess
Primary Format Universal (JPEG, WAV, BMP) JPEG optimized
Technique LSB substitution DCT coefficient modification
Encryption Rijndael-128 included Optional password
Detection Resistance Good Very Good (harder to detect)
Capacity 20-30% 15-25% (but more secure)

OutGuess Usage Examples

# BASIC EMBEDDING
outguess -e -d secret.txt original.jpg stego.jpg
# -e: Embed mode
# -d: Data file to hide
# Result: stego.jpg contains hidden data

# BASIC EXTRACTION
outguess -r -d recovered.txt stego.jpg
# -r: Read/extract mode
# Recovers hidden data to recovered.txt

# WITH ENCRYPTION/PASSWORD
outguess -e -p "MyPassword123" -d secret.txt original.jpg stego.jpg
outguess -r -p "MyPassword123" -d recovered.txt stego.jpg

# GET FORMAT INFORMATION
outguess -i stego.jpg
# Shows capacity, compression quality, DCT information

# QUALITY/COMPRESSION OPTIONS
outguess -e -q 75 -d secret.txt original.jpg stego.jpg
# -q: JPEG quality (higher = less distortion but larger file)

OutGuess vs Steghide Practical Comparison

  • Steghide: Better for multi-format deployment, easier to use
  • OutGuess: Better for JPEG-specific scenarios, more resistant to detection
  • Detection Difficulty: OutGuess slightly harder to detect statistically
  • Speed: Both comparable, OutGuess slightly faster
  • Capacity: Similar, depends on image size and compression
  • Recommendation: Use OutGuess for critical data, Steghide for convenience

NTFS Alternate Data Streams (ADS)

Hiding files within files on Windows systems

What are Alternate Data Streams?

NTFS ADS: Hidden file forks that allow multiple data streams in single file

  • Invisible by Default: Normal tools don't show ADS
  • Data Retention: ADS survives file copying (within NTFS)
  • Capacity: Unlimited (theoretically, limited by disk space)
  • Attribution: Malware hiding technique, not immediately obvious
  • Original File Unaffected: Can't tell main file contains stream data

NTFS ADS Creation & Extraction

# HIDE TEXT MESSAGE IN FILE
echo "Welcome to the college" > readme.txt:secret.txt
# Main file: readme.txt (contains original content)
# Hidden stream: secret.txt (invisible)

# VIEW MAIN FILE CONTENT
type readme.txt
# Output: Only shows main stream content
# Hidden stream NOT visible

# VIEW HIDDEN STREAM
type readme.txt:secret.txt
# Output: "Welcome to the college"

# CREATE HIDDEN EXECUTABLE
copy malware.exe readme.txt:evil.exe
# readme.txt appears normal (original size)
# evil.exe hidden in alternate stream

# HIDE ENTIRE DIRECTORY IN STREAM
dir C:\Confidential > readme.txt:folder_list.txt
# Directory listing hidden in stream

# LIST ALL STREAMS (WINDOWS)
dir /R readme.txt
# Shows main stream and all hidden streams

# DETECT STREAMS WITH TOOLS
dir /R
# Built-in Windows command (streams visible as)
# readme.txt:secret.txt:$DATA

# REMOVE HIDDEN STREAM
type readme.txt > temp.txt && del readme.txt && ren temp.txt readme.txt
# Deletes all alternate streams (copies main stream only)

NTFS ADS Attack Scenario: Backdoor Installation

Objective: Hide Backdoor ▼

Attacker has command execution on Windows system

Wants to hide backdoor executable where it won't be detected

Standard folders (System32, Program Files) monitored

Step 1: Upload Malware ▼
C:\> backdoor.exe (100KB file) # Attacker uploads via file upload vulnerability
Step 2: Hide in Stream ▼
copy C:\Windows\System32\notepad.exe C:\Windows\System32\notepad.exe:hidden.exe copy backdoor.exe C:\Windows\System32\notepad.exe:hidden.exe # Hidden in legitimate system file
Step 3: Create Persistence ▼
reg add "HKCU\Software\Microsoft\Windows\CurrentVersion\Run" \ /v "Update" /t REG_SZ /d "notepad.exe:hidden.exe" # Registry entry executes hidden backdoor on next reboot
Step 4: Deletion & Cleanup ▼
del backdoor.exe # Original file deleted # notepad.exe appears normal size (~200KB) # Hidden stream not visible to administrators
Step 5: Persistence Achieved ▼

On system reboot: notepad.exe:hidden.exe executes automatically

Backdoor connects to C&C server for remote access

Administrator runs dir/type commands - sees nothing unusual

🔴 NTFS ADS RISKS:
  • Malware hiding and execution without visibility
  • Rootkit installation in hidden streams
  • Backdoor persistence mechanisms
  • Worm propagation through network shares
  • Data theft - databases hidden in system files
✓ DETECTION & REMOVAL: Use Streams tool, dir /R command, or PowerShell: (Get-Item "file.txt" -Stream *)

White Space Steganography

Hiding data in text file spacing and invisible characters

White Space Stego Basics

Principle: Embed data in invisible characters (spaces, tabs, newlines)

  • Target: Text files, documents, source code
  • Invisible: When viewed normally, document appears unchanged
  • Encoding: Each space/tab represents 1 bit (space=0, tab=1)
  • Detection: Select-all (Ctrl+A) reveals extra whitespace
  • Tools: Snow, SNOW32.exe (Windows implementation)

White Space Steganography with SNOW

# HIDE SECRET MESSAGE IN TEXT FILE
snow -C -m "This is secret information" -p "password123" input.txt output.txt
# -C: Compress the message
# -m: Message to hide
# -p: Passphrase protection
# Result: output.txt has hidden message in whitespace

# EXTRACT HIDDEN MESSAGE
snow -C -p "password123" output.txt
# Output: "This is secret information"
# Only valid with correct passphrase

# ADVANCED: HIDE ENTIRE FILE IN DOCUMENT
snow -C -f secret.zip -p "pass456" cover.txt output.txt
# Embeds entire ZIP file in document whitespace

# ESTIMATE CAPACITY
snow -c cover.txt
# Shows how much data can be embedded

# BATCH PROCESS FILES
for file in *.txt; do
  snow -C -m "Tagged by XYZ" -p "shared_key" "$file" "${file}.stego"
done

White Space Detection Methods

Detection Method Effectiveness Effort Required
Visual Inspection (Select-All) Low (only obvious cases) Minimal
Entropy Analysis Medium (statistical) Moderate (scripting)
Forensic Tool Scanning High (dedicated tools) Moderate (tool setup)
Byte-Level Analysis Very High (definitive) High (manual review)

Practical Applications

  • Insider Threat: Employee hides confidential documents in email messages
  • C&C Communication: Attacker hides commands in public documents/comments
  • Data Exfiltration: Hidden metadata in source code commits
  • Supply Chain Attack: Malicious payload embedded in software documentation
  • Plausible Deniability: Sender claims document is unmodified

Image Encoding for Data Exfiltration

Converting sensitive data to base64/encoded formats in images

Image Encoding Techniques

  • Base64 Encoding: Convert binary to ASCII text (50% size increase)
  • Hexadecimal: Convert to hex format (double size)
  • LSB Replacement: Embed data in least significant bits
  • Metadata Injection: Hide data in EXIF, IPTC, XMP fields
  • Polyglot Files: Create files valid as both JPG and data

Base64 Image Data Encoding

# ENCODE FILE TO BASE64 FOR EMBEDDING
base64 confidential.pdf > encoded.txt
# Result: Text-based representation of binary file

# EMBED IN IMAGE METADATA
exiftool -Comment="$(cat encoded.txt)" photo.jpg output.jpg
# Hides base64 data in EXIF comment field

# EXTRACT FROM IMAGE
exiftool -Comment output.jpg > extracted.txt
base64 -d extracted.txt > recovered.pdf

# CREATE POLYGLOT JPG + ZIP FILE
cat image.jpg data.zip > output.jpg
# Valid as both JPG image and ZIP archive

# VERIFY POLYGLOT FUNCTIONALITY
file output.jpg  # Shows as JPEG
unzip output.jpg # Extracts ZIP contents

# ENCODE ENTIRE DATABASE
gzip -9 database.sql > database.sql.gz
base64 database.sql.gz > payload.txt
# Embed payload.txt in image EXIF
✓ ADVANTAGES:
  • Text-based format bypasses many filters
  • EXIF metadata appears normal to viewers
  • Multiple encoding layers increase detection difficulty
  • Standard command-line tools (no special software needed)

Steganography Detection & Analysis

Identifying covert data in files

Detection Challenges

  • Perfect Steganography: Undetectable if executed well
  • False Positives: Legitimate high-quality images trigger alerts
  • Multiple Layers: Combining stego + encryption + encoding
  • Tool-Specific: Different tools use different techniques
  • No Signature: Unlike malware, no hash/signature to match

Detection Methods & Tools

Statistical Analysis

  • Chi-Square Test: Analyze byte distribution deviations
  • Entropy Analysis: Steganography typically increases entropy
  • Histogram Inspection: Unusual pixel value patterns
  • DCT Analysis: For JPEG steganography detection

Forensic Tools

  • Stegdetect: Detects JSteg, OutGuess, Steghide
  • Stegowatch: Real-time stego file detection
  • Exiftool: Extract and analyze metadata
  • Binwalk: Identify file signatures and embedded data

NTFS ADS Detection

  • dir /R: Built-in Windows command shows streams
  • Streams (SysinternalS): List and manage ADS
  • PowerShell: Get-Item "file" -Stream *
  • Forensic Tools: Autopsy, FTK, EnCase detect streams

Malware Classification & Types

Understanding different malware categories and characteristics

Malware Definition & Purpose

Malware: Malicious software designed to damage, disable, or compromise systems

  • Damage Systems: Corrupt files, delete data, cause hardware failure
  • Steal Information: Passwords, credit cards, personal data
  • Commit Fraud: Financial theft, identity theft
  • Disruption: DDoS attacks, botnet participation
  • Espionage: Corporate/government intelligence gathering

Malware Types Comparison

Type Self-Replicating Primary Goal Speed of Spread
Virus Yes (needs host) Damage/Corruption Medium
Worm Yes (standalone) Rapid spread Very Fast
Trojan No Backdoor access Manual delivery
Ransomware Rarely Encryption + Extortion Varies
Spyware Rarely Surveillance Stealth delivery
Keylogger No Keystroke capture Silent installation

Virus: Self-Replicating but Host-Dependent

  • Requirement: Must attach to executable files (.exe, .dll, .sys, .com)
  • Propagation: Spreads only when infected file executed
  • Example: File infector virus, macro virus in Word/Excel
  • Slower than Worms: Depends on user executing infected file
  • Detection: Signature scanning, heuristic analysis

Worm: Self-Propagating & Independent

  • No Host File Needed: Spreads standalone via network vulnerabilities
  • Rapid Infection: Internet worms infect thousands/millions in hours
  • Famous Examples: Nimda, Code Red, WannaCry (hybrid)
  • Network Scanning: Worm scans for vulnerable systems automatically
  • Zero-Day Exploitation: Unpatched vulnerabilities = rapid spread

Trojan: Disguised Backdoor Access

  • Not Self-Replicating: Requires manual distribution
  • Deceptive Appearance: Appears as legitimate program (game, tool, update)
  • Backdoor Installation: Opens remote access channel for attacker
  • RAT Component: Remote Administration Tool for control
  • Persistence: Stays hidden and connects to C&C server

Ransomware: Encryption & Extortion

  • Encryption: Encrypts files with attacker-controlled key (usually RSA-2048)
  • Ransom Demand: Display ransom message with payment instructions
  • Bitcoin Payments: Untraceable cryptocurrency for payment
  • Decryption: Victims sometimes get decryption key after payment (not guaranteed)
  • Examples: WannaCry, NotPetya, REvil, LockBit

Malware Distribution Methods

How attackers deliver malware to targets

Primary Delivery Vectors

  • Email Attachments: Most common delivery method
  • Messaging Apps: WhatsApp, Telegram, Facebook Messenger links
  • USB/Removable Media: Infected drives planted in public
  • Malicious Websites: Drive-by download, exploit kit hosting
  • Software Updates: Fake/compromised update executables
  • Macro Viruses: Embedded in Office documents

Phishing Email Attack Chain

Step 1: Social Engineering ▼

Attacker researches target company and employees

Identifies accounts or departments with sensitive access

Creates believable scenario (security alert, compliance update, etc.)

Step 2: Email Crafting ▼

Subject: "URGENT: Account Verification Required"

From: spoofed@company.com (looks legitimate)

Attachment: "AccountVerification.exe" or macro-enabled .DOCX

Step 3: User Interaction ▼

Employee receives email during work hours

Feels urgency (compliance deadline, security issue)

Downloads and opens attachment (malware delivered)

Step 4: Execution ▼

Malware executable runs with user privileges

Establish persistence (registry, scheduled task, etc.)

Download additional payloads (RAT, keylogger, etc.)

Step 5: Compromise ▼

System becomes infected and attacker gains access

Lateral movement to other systems

Data theft or further exploitation

Why These Methods Work

  • User Trust: Appears to come from trusted sources
  • Urgency: Creates psychological pressure to act quickly
  • Low Patch Management: 70% of corporate systems have unpatched vulnerabilities
  • Poor Email Filtering: Many filters don't catch sophisticated phishing
  • User Familiarity: People more likely to click known file types

Malware Attack Vectors & Exploitation

Technical methods for malware infection and execution

Common Attack Vector Vulnerabilities

  • Outdated Web Browsers: Unpatched browser vulnerabilities (Flash, etc.)
  • Browser Plugins: Java, Adobe Reader, ActiveX exploits
  • Insecure Patch Management: Delayed Windows/application updates
  • Open Wireless Networks: Man-in-the-middle interception and injection
  • Instant Messengers: File sharing features without proper validation
  • Email Clients: HTML rendering vulnerabilities, attachment handling
  • PDF Readers: Embedded exploits in malicious PDFs
⚠️ ZERO-DAY VULNERABILITIES:
  • Definition: Exploits for unknown vulnerabilities with no available patch
  • Timeline: Exploit exists before vendor is aware or patch released
  • Impact: Complete system compromise with no defense
  • Marketability: Zero-day exploits sold on dark web for $100K-$1M+
  • APT Usage: Nation-states and advanced attackers primary users
✓ MITIGATION STRATEGIES:
  • Enable automatic Windows updates and patches
  • Use latest browser versions (Chrome auto-updates by default)
  • Disable legacy technologies (Java, Flash, ActiveX)
  • Install security software (antivirus + anti-malware)
  • Network-level protection (IDS/IPS, next-gen firewalls)
  • User education on phishing and malware dangers

Trojans & Remote Access Trojans (RAT)

Understanding backdoor mechanisms and remote control

What is a Trojan?

Trojan Horse: Malware disguised as legitimate software with hidden malicious code

  • Not Self-Replicating: Doesn't spread on its own (requires user download)
  • Deceptive Appearance: Looks like game, tool, or utility
  • Hidden Payload: Contains backdoor, keylogger, or data stealer
  • Privilege Level: Runs with same privileges as user who executed it
  • Persistence: Often installs into startup folders or registry

Remote Access Trojan (RAT) Capabilities

  • Remote Command Execution: Execute arbitrary commands on victim machine
  • File Transfer: Upload/download files to/from infected system
  • Screen Capture: Steal desktop screenshots in real-time
  • Keylogging: Record all keyboard input (passwords, emails, etc.)
  • Webcam Access: Activate camera for surveillance (often covertly)
  • Microphone: Record audio from system
  • Process Management: View, kill, or restart running processes
  • Registry Access: Read/modify Windows registry
  • Reverse Shell: Interactive command shell access

Famous RAT Examples

  • T88: Demonstration tool (often used for educational purposes)
  • Dark Comet: Common RAT with GUI control interface
  • Poison Ivy: Older but effective RAT (still used)
  • Metasploit Meterpreter: Professional penetration testing RAT
  • AsyncRAT: Modern Open-Source RAT (actively used)
  • LimeRAT: Recent RAT discovered in malware campaigns
🔴 TROJAN INDICATORS OF COMPROMISE (IoC):
  • Unexpected background network activity (high bandwidth usage)
  • Unknown processes running in Task Manager
  • Slow system performance / high CPU/memory usage
  • Unexpected login attempts or failed authentications
  • New user accounts discovered on system
  • Suspicious registry entries or startup programs
  • Files mysteriously modified or deleted
  • Firewall showing unknown outbound connections

Cryptographic Foundations & Key Concepts

Understanding encryption, hashing, and cryptographic protocols

Symmetric vs Asymmetric Encryption

Aspect Symmetric Asymmetric
Algorithm Type Single key for encryption/decryption Public key (share) + Private key (secret)
Key Exchange Difficult - key distribution problem Easy - use public key, keep private key secret
Speed Very fast (100+ MB/sec) Slow (1-10 MB/sec)
Common Algorithms AES-256, DES, 3DES, RC4 RSA, ECC, ElGamal
Primary Use Bulk data encryption Key exchange, digital signatures
Vulnerability to QC Resistant to Quantum Computing Vulnerable to quantum algorithms

Hashing Functions & Digital Signatures

Hash Function: Converts input data of any size to fixed-size output (digest)

  • Properties: Deterministic, fast, avalanche effect, collision-resistant
  • One-Way: Cannot reverse hash to get original data
  • Common Algorithms: MD5 (128-bit), SHA-1 (160-bit), SHA-256 (256-bit), SHA-512 (512-bit)
  • Digital Signature Use: Hash+sign prevents tampering and proves authenticity

Example: A certificate with compromised hash (MD5) can be forged by attackers

Hash Collision Attacks

  • MD5 Collision: Found in 2004, two different inputs produce same hash
  • SHA-1 Collision: Google Shattered project (2017) demonstrated collision
  • Certificate Forgery: With hash collision, attacker can create valid cert for any domain
  • Mitigation: Use SHA-256 or stronger for all cryptographic operations

Man-in-the-Middle Attack with Weak Encryption

Scenario: Attacker intercepts SSL/TLS connection and downgrades encryption

Step 1: Connection Initiation ▼

Client: "Hello Server, I want secure connection (TLS 1.3)"

Attacker (MITM): Intercepts handshake

Attacker→Server: "Connected client wants TLS 1.0" (downgrade)

Step 2: Cipher Suite Negotiation ▼

Server: Accepts TLS 1.0, offers cipher suite: RC4-128

Client→Attacker: Agrees to RC4-128

Attacker Status: Can now break RC4 (known attacks)

Step 3: Key Exchange ▼

Client & Server: Exchange RC4 encryption keys

Attacker: Captures both key exchange messages

Attacker Capability: Brute-force RC4 in minutes to recover session key

Step 4: Data Decryption ▼

Client: Sends login credentials encrypted with RC4

Attacker: Uses recovered key to decrypt credentials

Attacker: username: admin, password: Finance2024!

Step 5: Full Compromise ▼

Attacker: Uses recovered credentials

Impact:

  • Account takeover
  • Data theft/modification
  • Further system compromise
  • Lateral movement to other systems

Cryptographic Security Best Practices

Algorithm Selection

  • ✓ Use AES-256 for symmetric encryption (no known practical attacks)
  • ✓ Use RSA-2048 or ECC-256 for asymmetric operations
  • ✓ Use SHA-256 or SHA-512 for hashing (never MD5 or SHA-1)
  • ✗ Avoid: RC4, DES, MD5, SHA-1, deprecated algorithms

Protocol Enforcement

  • ✓ Enforce TLS 1.2 minimum (TLS 1.3 preferred)
  • ✓ Disable SSL negotiation (SSL 3.0, TLS 1.0/1.1)
  • ✓ Use Perfect Forward Secrecy (PFS) cipher suites
  • ✓ Implement HSTS (HTTP Strict Transport Security)

Key Management

  • ✓ Store private keys in Hardware Security Modules (HSM)
  • ✓ Rotate keys annually or on compromise
  • ✓ Use strong key derivation functions (PBKDF2, Argon2)
  • ✓ Implement zero-knowledge architectures

Kerberos Authentication Protocol & Attacks

Understanding the foundation of enterprise authentication and its vulnerabilities

Kerberos Protocol Architecture

  • Authentication Server (AS): Issues Ticket Granting Tickets (TGT)
  • Ticket Granting Server (TGS): Issues service tickets for resource access
  • Key Distribution Center (KDC): Contains AS and TGS (typically on Domain Controller)
  • Application Server: Resources accessed by authenticated users
  • Encryption: All tickets encrypted with user password hash

Kerberos Authentication Flow

Complete 5-Step Kerberos Process

Phase 1: TGT Request (AS-REQ) ▼

Client Action:

User logs in: username "admin", password "SecureP@ss123"

Client→AS:

  • Sends username in PLAIN TEXT
  • Creates timestamp encrypted with user's password hash
  • Proof of possession (user knows password)

Attack Point: Password hash can be cracked offline (AS-REP roasting)

Phase 2: TGT Response (AS-REP) ▼

AS→Client: Returns TGT (encrypted with KDC password)

  • TGT contains: user identity, session key, validity period
  • Encrypted with TDC's master key (hidden from user)
  • User's session key: encrypted with user password hash
  • User cannot decrypt TGT (only KDC can)

Attack Point: TGT reuse across multiple service requests

Phase 3: Service Ticket Request (TGS-REQ) ▼

User wants to access: File Server (FILESERVER$)

Client→TGS:

  • Sends TGT (proves prior authentication)
  • Requests service ticket for FILESERVER
  • Includes authenticator (timestamp + user key)

Attack Point: Kerberoasting - crack service account password

Phase 4: Service Ticket Response (TGS-REP) ▼

TGS→Client: Returns service ticket

  • Service ticket encrypted with service account password hash
  • Contains user identity and authorization data
  • User receives session key for service communication

Threat: Service tickets can be captured and cracked offline

Phase 5: Service Access (AP-REQ/AP-REP) ▼

Client→FileServer: Presents service ticket

FileServer: Validates ticket with TGS party confirmation

Access Granted: User can now access files

Attack Point: Pass-the-ticket (PT-T) - use captured ticket directly

🔴 CRITICAL KERBEROS ATTACKS:

1. Pass-The-Ticket (PTT):

  • Attacker captures Kerberos ticket from memory
  • Injects ticket into another process
  • Immediately accesses resources without password
  • Detection: Ticket reuse from unexpected machines/times

2. Golden Ticket:

  • Attacker compromises KDC master key
  • Forges Kerberos TGT with Domain Admin privileges
  • Complete forest compromise - permanent access
  • Detection: Ticket validity exceeds 10-hour default

3. Kerberoasting:

  • Request service tickets for all service accounts
  • Crack service account passwords offline
  • Often service passwords weak/unchanged for years
  • Detection: Spike in TGS requests from single user

4. AS-REP Roasting:

  • Target accounts with "Do not require Kerberos pre-authentication"
  • Request TGT without password proof
  • Crack TGT offline (contains user password hash)
  • Detection: AS-REQ for accounts without pre-auth

Kerberos Security Hardening

User Account Protection

  • ✓ Enforce strong passwords (20+ characters minimum)
  • ✓ Enforce password complexity requirements
  • ✓ Implement password history (prevent reuse)
  • ✓ Set password age limits (30-60 days)
  • ✓ Monitor failed login attempts (alert after 5 failures)

  • ✓ Never use user accounts for services

  • ✓ Disable "Pre-authentication required" is false
  • ✓ Monitor service account elevation attempts

Ticket & KDC Protection

  • ✓ Limit TGT lifetime to 8-10 hours maximum
  • ✓ Implement ticket renewal checks
  • ✓ Monitor KDC logs for anomalous ticket requests
  • ✓ Use KRBTGT account password rotation (every 180 days)
  • ✓ Enable Kerberos armoring (FAST) for clients
